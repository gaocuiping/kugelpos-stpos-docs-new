# Sync Service データ同期 アーキテクチャー解説書

## 1. システム全体構成

### 1.1 システム概要

Kugelpos POSシステムにおけるSync Serviceは、クラウド環境とエッジ環境（店舗）間でデータの双方向同期を実現する中核的なサービスです。システムは大きく2つの環境に分かれており、それぞれが独立したサービス群を持ちながら、Sync Serviceを通じて双方向のデータ同期を行います。

### 1.3 同期の特徴

- **双方向同期**: クラウドとエッジ間で双方向のデータ同期を実現
- **マルチデバイス対応**: 1店舗に複数台のエッジ端末（POS内蔵型、専用エッジサーバー）が存在する構成をサポート
- **自動フェイルオーバー**: ネットワーク障害時もエッジ環境で業務継続可能
- **差分同期**: 効率的な差分データの同期により、ネットワーク帯域を最適化
- **P2Pファイル共有**: 同一店舗内のエッジ端末間でマスターデータ予約ファイルを共有し、クラウド負荷を軽減
- **ファイル収集**: エッジ環境のアプリケーションログやシステムファイルをzip形式で圧縮収集
- **データアクセス制限**: Syncサービスは自身のデータベースのみ直接アクセス可能、他サービスのデータはAPIを通じてアクセス
- **テナント分離**: エッジ端末情報を含む全データをテナント別DBで管理し、完全な分離を実現

## 2. データフローパターン

### 2.1 クラウド → エッジ（マスターデータ）

マスターデータは通常クラウドで管理され、エッジ環境に配信されます。

```mermaid

        direction LR
        CMD2[Cloud Master Data] --> CS2[Cloud Sync]
    end

    subgraph Edge["Edge Environment"]
        direction LR
        ES2[Edge Sync] --> EMD2[Edge Master Data]
    end

    CS2 -->|Master Data<br/>Products, Payment Methods<br/>Tax Rules, Staff| ES2

    style Cloud fill:#e1f5fe
    style Edge fill:#fff3e0
```

**対象データ:**
- **Products（商品情報）**: 商品マスター、価格、在庫情報
- **Payment Methods（決済方法）**: 利用可能な決済手段の設定
- **Tax Rules（税制ルール）**: 税率、税計算ロジック
- **Staff（スタッフ情報）**: 従業員マスター、権限設定

**同期フロー:**

```mermaid
sequenceDiagram
    participant EdgeSync as Edge Sync
    participant CloudSync as Cloud Sync
    participant CloudMasterData as Cloud Master-Data
    participant EdgeMasterData as Edge Master-Data

    Note over EdgeSync,CloudSync: 定期ポーリング（30-60秒間隔）

    EdgeSync->>CloudSync: 更新確認リクエスト<br/>(data_type=master_data, last_sync_timestamp)

    CloudSync->>CloudMasterData: 差分データ取得<br/>(updated_at > last_sync_timestamp)
    CloudMasterData-->>CloudSync: 変更されたマスターデータ

    alt データあり
        CloudSync->>CloudSync: データ圧縮
        CloudSync-->>EdgeSync: マスターデータ差分<br/>(products, payment_methods, tax_rules, staff)

        EdgeSync->>EdgeSync: データ解凍・検証

        EdgeSync->>EdgeMasterData: データ適用<br/>(Dapr Service Invocation)
        EdgeMasterData->>EdgeMasterData: DB更新
        EdgeMasterData-->>EdgeSync: 適用完了

        EdgeSync->>CloudSync: 同期完了通知
        CloudSync-->>EdgeSync: 確認
    else データなし
        CloudSync-->>EdgeSync: 変更なし
    end
```

### 2.7 マスターデータ整合性チェックフロー

マスターデータ同期では、データの完全性と整合性を保証するため、段階的なチェックメカニズムを実装しています。

#### 2.7.1 一括同期の整合性チェックフロー

```mermaid
sequenceDiagram
    participant EdgeSync as Edge Sync
    participant CloudSync as Cloud Sync
    participant CloudMasterData as Cloud Master-Data
    participant EdgeMasterData as Edge Master-Data
    participant EdgeDB as Edge DB

    Note over EdgeSync,CloudSync: 一括同期リクエスト

    EdgeSync->>CloudSync: 一括同期要求<br/>(data_type=master_data, sync_type=bulk)

    CloudSync->>CloudMasterData: 全件データ取得
    CloudMasterData-->>CloudSync: 全マスターデータ

    CloudSync->>CloudSync: チェックサム計算<br/>(SHA-256, 主キー昇順ソート)
    CloudSync->>CloudSync: レコード件数カウント

    CloudSync-->>EdgeSync: 圧縮データ + チェックサム + 件数

    Note over EdgeSync: データ受信整合性チェック

    EdgeSync->>EdgeSync: データ解凍
    EdgeSync->>EdgeSync: チェックサム再計算

    alt チェックサム一致
        EdgeSync->>EdgeSync: ✓ チェックサム検証成功

        Note over EdgeSync,EdgeDB: データ永続化整合性チェック

        EdgeSync->>EdgeMasterData: データ適用要求<br/>(shadow table方式)
        EdgeMasterData->>EdgeDB: 新バージョンデータ投入<br/>(active=false)
        EdgeDB-->>EdgeMasterData: 投入完了
        EdgeMasterData->>EdgeDB: バージョン切替<br/>(トランザクション)
        EdgeDB-->>EdgeMasterData: 切替完了
        EdgeMasterData-->>EdgeSync: 適用完了

        EdgeSync->>EdgeDB: レコード件数確認
        EdgeDB-->>EdgeSync: 実測件数

        alt 件数一致
            EdgeSync->>EdgeSync: ✓ レコード件数検証成功
            EdgeSync->>CloudSync: 同期完了通知
            CloudSync-->>EdgeSync: 確認
        else 件数不一致
            EdgeSync->>EdgeDB: ロールバック実行
            EdgeSync->>EdgeSync: リトライ判断<br/>(最大3回)
            alt リトライ可能
                EdgeSync->>CloudSync: 再同期要求
            else リトライ上限
                EdgeSync->>EdgeSync: ✗ エラーログ出力<br/>アラート送信
            end
        end

    else チェックサム不一致
        EdgeSync->>EdgeSync: データ破棄
        EdgeSync->>EdgeSync: リトライ判断<br/>(最大3回)
        alt リトライ可能
            EdgeSync->>CloudSync: 再同期要求
        else リトライ上限
            EdgeSync->>EdgeSync: ✗ エラーログ出力<br/>アラート送信
        end
    end
```

**チェックポイント:**
1. **データ受信整合性**: チェックサム検証により通信途中のデータ破損・欠損を検出
2. **データ永続化整合性**: レコード件数検証によりDB投入の完全性を保証

#### 2.7.2 差分同期の整合性チェックフロー

```mermaid
sequenceDiagram
    participant EdgeSync as Edge Sync
    participant CloudSync as Cloud Sync
    participant CloudMasterData as Cloud Master-Data
    participant EdgeMasterData as Edge Master-Data
    participant EdgeDB as Edge DB

    Note over EdgeSync,CloudSync: 差分同期リクエスト（30-60秒間隔）

    EdgeSync->>CloudSync: 差分同期要求<br/>(last_sync_timestamp)

    CloudSync->>CloudMasterData: 差分データ取得<br/>(updated_at > last_sync_timestamp)
    CloudMasterData-->>CloudSync: 変更レコード群

    CloudSync->>CloudSync: 差分チェックサム計算<br/>(操作タイプ別, updated_at昇順)
    CloudSync->>CloudSync: 操作タイプ別件数カウント<br/>(INSERT/UPDATE/DELETE)

    CloudSync-->>EdgeSync: 差分データ + チェックサム + 件数内訳

    Note over EdgeSync: データ受信整合性チェック

    EdgeSync->>EdgeSync: データ解凍
    EdgeSync->>EdgeSync: 差分チェックサム再計算

    alt チェックサム一致
        EdgeSync->>EdgeSync: ✓ チェックサム検証成功

        Note over EdgeSync,EdgeDB: データ永続化整合性チェック

        EdgeSync->>EdgeMasterData: 差分データ適用要求
        EdgeMasterData->>EdgeDB: INSERT/UPDATE/DELETE実行
        EdgeDB-->>EdgeMasterData: 実行結果<br/>(処理件数含む)
        EdgeMasterData-->>EdgeSync: 適用完了<br/>(処理件数内訳)

        EdgeSync->>EdgeSync: 操作タイプ別件数照合

        alt 件数一致
            EdgeSync->>EdgeSync: ✓ レコード件数検証成功
            EdgeSync->>CloudSync: 同期完了通知
            CloudSync-->>EdgeSync: 確認

            Note over EdgeSync: 5分後にバージョン検証実行
        else 件数不一致
            EdgeSync->>EdgeDB: ロールバック実行
            EdgeSync->>EdgeSync: リトライ判断<br/>(最大3回)
        end

    else チェックサム不一致
        EdgeSync->>EdgeSync: データ破棄
        EdgeSync->>EdgeSync: リトライ判断<br/>(最大3回)
    end
```

**チェックポイント:**
1. **データ受信整合性**: 差分チェックサム検証（操作タイプ別）
2. **データ永続化整合性**: 操作タイプ別レコード件数検証（INSERT/UPDATE/DELETE）
3. **更新完全性**: バージョン検証（5分間隔、次のフローで説明）

#### 2.7.3 バージョン検証と補完同期フロー

```mermaid
sequenceDiagram
    participant EdgeSync as Edge Sync
    participant CloudSync as Cloud Sync
    participant EdgeDB as Edge DB
    participant CloudDB as Cloud DB

    Note over EdgeSync: 5分間隔の定期バージョン検証

    EdgeSync->>CloudSync: バージョン情報要求<br/>(data_type)
    CloudSync->>CloudDB: 最新バージョン取得
    CloudDB-->>CloudSync: cloud_latest_version

    CloudSync-->>EdgeSync: 最新バージョン情報

    EdgeSync->>EdgeDB: エッジ最新バージョン取得
    EdgeDB-->>EdgeSync: edge_latest_version

    EdgeSync->>EdgeSync: バージョン連続性確認<br/>(欠番検出)

    alt バージョン一致 & 欠番なし
        EdgeSync->>EdgeSync: ✓ バージョン整合性OK
        Note over EdgeSync: 次回検証まで待機
    else バージョン差異あり & 欠番なし
        EdgeSync->>EdgeSync: 最新バージョン取得が必要
        EdgeSync->>CloudSync: 差分同期要求<br/>(正常処理)
    else 欠番検出 & 50件以下
        EdgeSync->>EdgeSync: 欠落バージョン特定<br/>(最大20件/回)

        Note over EdgeSync,CloudSync: 補完同期実行

        EdgeSync->>CloudSync: POST /api/v1/sync/backfill<br/>{missing_versions: [47, 156, 342]}
        CloudSync->>CloudDB: 指定バージョンデータ取得
        CloudDB-->>CloudSync: バージョン別データ
        CloudSync-->>EdgeSync: 補完データ

        EdgeSync->>EdgeDB: バージョンデータ投入
        EdgeDB-->>EdgeSync: 投入完了

        EdgeSync->>EdgeSync: 欠落バージョンリストから削除

        EdgeSync->>CloudSync: 補完完了通知
        CloudSync-->>EdgeSync: 確認
    else 欠番検出 & 50件超過
        EdgeSync->>EdgeSync: ✗ 一括同期推奨
        EdgeSync->>EdgeSync: アラート送信<br/>(VERSION_GAP_EXCEEDED)
        Note over EdgeSync: 手動または自動で一括同期実行
    end
```

**バージョン検証の目的:**
- **更新漏れの検出**: ネットワーク障害等で取得できなかったバージョンを特定
- **自動復旧**: 欠落バージョンを補完同期で自動取得
- **一括同期判断**: 欠落が多い場合は一括同期への切り替えを推奨

**補完同期の特徴:**
- 通常の差分同期と並行実行可能
- `last_sync_timestamp`は更新しない（現在の同期状態を維持）
- 処理件数制限：20バージョン/回（システム負荷考慮）

### 2.8 エッジ端末間P2Pファイル共有（マスターデータ予約ファイル）

同一店舗内の複数エッジ端末間でマスターデータ予約ファイルを共有し、クラウドへの通信負荷を軽減します。

```mermaid
flowchart TB
    subgraph Cloud["Cloud Environment"]
        CS_P2P[Cloud Sync<br/>File Holding Status Management]
    end

    subgraph Store["Store (Multiple Edge Devices)"]
        direction LR
        Edge1[Edge Server<br/>device_type=edge<br/>is_p2p_seed=true, priority=0<br/><b>役割: プライマリP2Pシード</b><br/>クラウドから最初にダウンロード<br/>全POS端末へP2P配信]
```

**フロー:**
1. **Edge Server (priority=0)** がクラウドに予約ファイル情報を問い合わせ
2. クラウドが「他のエッジ端末が保持していない」ことを通知
3. **Edge Server** がクラウドからファイルをダウンロード（最優先シード）
4. **Edge Server** がダウンロード完了をクラウドに通知
5. **POS-001 (priority=1)** が同じファイルを問い合わせ
6. クラウドが「Edge Serverが保持している（priority=0）」ことを通知
7. **POS-001** が**Edge Server**からP2P転送でファイル取得
8. **POS-001** がダウンロード完了をクラウドに通知
9. **POS-002 (priority=99)** が同じファイルを問い合わせ
10. クラウドが「Edge Server (priority=0), POS-001 (priority=1)が保持している」ことを通知
11. **POS-002** が優先度順に**Edge Server → POS-001**からP2P転送で取得
12. **POS-002** がダウンロード完了をクラウドに通知

**P2Pダウンロード優先順位:**
- **第1優先**: Edge Server (device_type=edge, priority=0) - 専用サーバー、最も高性能で安定
- **第2優先**: POS-001 (device_type=pos, priority=1) - Edge Server不在時やバックアップとして機能
- **第3優先**: クラウドからダウンロード - すべてのシードで失敗した場合（シード端末のみ）

**デバイス役割の詳細:**

| device_type | is_p2p_seed | priority | 役割 | 責務 |
|-------------|-------------|----------|------|------|
| edge | true | 0 | プライマリP2Pシード（専用サーバー） | クラウドから最初にマスタファイルをダウンロードし、店舗内の全POS端末へP2P配信。高可用性・高性能なサーバー機 |
| pos | true | 0-9 | P2Pシード兼POS端末 | Edge Server不在時、代表POS端末がシード機能を担当。Priority 0=プライマリ、1-9=セカンダリ |
| pos | false | 99 | 通常のPOS端末 | シード端末からP2P受信のみ。取引処理と販売業務に専念 |

**P2Pダウンロード優先順位:**
1. **Edge Server (priority=0)**: 専用エッジサーバーが存在する場合、最優先でクラウドからダウンロード
2. **POS (priority=0)**: Edge Server不在時、代表POS端末がプライマリシードとして機能
3. **POS (priority=1-9)**: セカンダリシードとしてフォールバック機能を提供

#### 2.9.4 P2Pフォールバック戦略

**フォールバックの優先順位:**
1. **優先度0のP2Pシード** (priority=0): プライマリシード
2. **優先度1-9のP2Pシード** (priority=1-9): セカンダリシード
3. **クラウドからのダウンロード**: すべてのシードで失敗した場合

**設計上の考慮点:**
- P2Pシード端末は複数設定可能（冗長性確保）
- 優先度は0-99の範囲で設定（0が最優先）
- 非シード端末（priority=99）はP2P受信のみ
- シード障害時は自動的に次の優先度のシードへフォールバック
- すべてのシードで失敗した場合、シード端末のみクラウドから取得可能

**将来の拡張:**
- アプリケーション更新機能では、`metadata`フィールドにコンテナレジストリ情報を格納
- P2P優先度制御の仕組みをコンテナイメージ配信にも適用可能

## 3. 同期メカニズム

### 3.1 差分同期メカニズム（Differential Sync）

最も一般的な同期方式で、変更があったデータのみを転送します。

**動作フロー:**
1. エッジ側のSync Serviceが30秒〜1分ごとにクラウドに更新確認を送信（ポーリング）
2. クラウド側が各サービスのAPIを通じて最後の同期タイムスタンプ以降の変更データを取得
3. 変更があったレコードのみを抽出（デルタデータ）
4. エッジ側が各サービスのAPIを通じてデルタデータを適用
5. 同期完了をクラウドに通知

**利点:**
- ネットワーク帯域の効率的な利用
- 高頻度での同期が可能
- リアルタイムに近い更新を実現

### 3.2 バルク同期メカニズム - スナップショット

マスターデータの完全同期が必要な場合に使用されます。

**動作フロー:**
1. エッジからフル同期リクエストを送信
2. クラウドが各サービスのAPIを通じて全レコードを取得
3. データを圧縮してスナップショットを作成
4. エッジ側が各サービスのAPIを通じて既存データを完全に置き換え
5. バルク同期完了を通知

**使用シーン:**
- 初期セットアップ時
- データ不整合の修復時
- 定期的なマスターデータのリフレッシュ

### 3.3 バルク同期メカニズム - ストリーム

大量のログやジャーナルデータの転送に使用されます。

**動作フロー:**
1. エッジからストリーム同期をリクエスト
2. クラウドがデータストリームを開始
3. データをチャンク単位で順次転送
4. エッジ側で受信したチャンクを順次追加
5. 全チャンク受信後、ストリーム完了を通知

**特徴:**
- メモリ効率的な大量データ転送
- 途中からの再開が可能
- ログやジャーナルなどの追記型データに最適

### 3.4 ファイル収集メカニズム

エッジ環境のファイル（アプリケーションログ含む）をzip形式で圧縮収集します。

**動作フロー:**
1. エッジ側の定期同期リクエストを送信
2. クラウド側が同期レスポンスに収集指示を含めて返却（オプション）
3. エッジ側が収集指示を受信した場合、ファイル収集処理を開始
4. 指定されたパス（ファイル/ディレクトリ）をzip形式で圧縮
5. 圧縮ファイルをクラウドの専用APIに送信
6. クラウド側でアーカイブを受信・保存し、収集完了をレスポンスで通知

### 6.1 同期処理ステートマシン


### 6.2 主要な状態

#### Idle（待機状態）
- 同期処理の開始を待つ基本状態
- 次の同期タイミングをスケジューリング
- リソースを最小限に保ちながら待機

#### CheckingUpdates（更新確認中）
- クラウドに対して更新の有無を確認
- 最後の同期タイムスタンプを基準に照会
- 更新がある場合はFetchingDataへ、なければIdleへ戻る

#### FetchingData（データ取得中）
- クラウドから実際のデータを取得
- ネットワーク帯域を考慮した効率的な転送
- 取得成功でApplyingChangesへ、失敗でErrorへ

#### ApplyingChanges（変更適用中）
- 取得したデータをローカルデータベースに適用
- トランザクション処理による一貫性の保証
- 適用成功でConfirmingSyncへ、失敗でErrorへ

#### ConfirmingSync（同期確認中）
- クラウドに同期完了を通知
- 同期ステータスを更新
- ファイル収集指示がある場合はCollectingFilesへ
- 確認完了でIdleへ戻り、失敗でErrorへ

#### CollectingFiles（ファイル収集中）
- ファイル収集指示を処理
- 指定されたパス（ファイル/ディレクトリ）を検証
- セキュリティチェック（ホワイトリスト確認）
- 収集準備完了でCompressingFilesへ、失敗でErrorへ

#### CompressingFiles（ファイル圧縮中）
- 収集対象ファイルをzip形式で圧縮
- 最大アーカイブサイズ制限の確認
- 圧縮完了でUploadingArchiveへ、失敗でErrorへ

#### UploadingArchive（アーカイブアップロード中）
- 圧縮ファイルをクラウドにアップロード
- チャンク分割による大容量ファイル対応
- アップロード完了でIdleへ、失敗でErrorへ

### 7.2 クラウド環境内通信（Cloud Sync ↔ Cloud Services）

クラウド環境では、**Pull型（API呼出）とPush型（Pub/Sub）が混在**します。

#### 7.2.1 通信方式一覧

| データ種別 | 通信方向 | 通信方式 | 理由 |
|-----------|---------|---------|------|

| ジャーナル取得 | Sync → journal | Dapr Service Invocation (API) | エッジから受信したジャーナルの確認 |
| トランザクションログ配信 | Sync → cart, terminal | **Dapr Pub/Sub (Push)** | エッジから受信したログを各サービスに配信 |
| ファイル保管 | Sync → Cloud Storage | Direct API | S3/GCS/Azure Blob等に直接保管 |

#### 7.4.3 データ圧縮と転送最適化

環境間通信では、ネットワーク帯域を節約するためにgzip圧縮を使用します。

**圧縮方式:**
- **圧縮アルゴリズム**: gzip (圧縮レベル6)
- **対象データ**: JSON形式のマスターデータ、トランザクションログ、ジャーナル
- **Content-Encoding**: `gzip` ヘッダーを使用
- **圧縮効果**: 平均60-80%のサイズ削減

**HTTPヘッダー:**
```
Authorization: Bearer {jwt_token}
Content-Type: application/gzip
Content-Encoding: gzip
```

## 9. 優先度制御と競合解決

### 9.1 同期優先度（現在の実装）

**処理方式:**
- すべてのデータ種別はFIFO（先入先出）方式で処理
- 同期間隔: 30秒〜1分（差分同期データタイプで統一）
- ファイル収集: オンデマンド（同期レスポンス連動）
- 特別な優先度制御なし（将来実装予定）

### 9.2 ファイル収集の優先度

**処理タイミング:**
- 定期同期レスポンスに収集指示が含まれる場合に実行
- 差分同期処理の後に実行
- エラー発生時はクラウドに状態通知

**優先度の特徴:**
- **オンデマンド実行**: 定期ポーリングは行わない
- **非阻害処理**: 通常の同期を妨げない設計
- **独立したエラー処理**: ファイル収集の失敗は通常同期に影響しない

### 9.3 競合解決戦略

#### Last Write Wins (LWW) - 現在の実装

**タイムスタンプベースの解決:**
- 最も新しいタイムスタンプのデータを採用
- シンプルで高速な解決方法
- すべての同期データタイプに適用
- ファイル収集は競合の概念なし（アーカイブファイルとして保存）

**実装詳細:**
- 各レコードに`updated_at`タイムスタンプを付与
- 同期時にタイムスタンプを比較
- 新しいデータで上書き
