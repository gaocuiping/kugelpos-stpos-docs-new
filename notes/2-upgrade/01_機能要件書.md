# Sync Service アプリ更新 - 機能要件書

## 1. 概要

### 1.1 目的

Kugelposシステムにおけるコンテナイメージ、スクリプト、設定ファイル等の配布物を一元管理し、クラウド・エッジ・POS端末間で効率的かつセキュアにバージョン管理・更新を実現する。

### 1.2 背景

- クラウド、エッジ、POS端末で動作する複数のマイクロサービスのバージョンを統一的に管理する必要がある
- ネットワーク障害時でもエッジ・POS端末が動作を継続できるよう、オフライン耐性が必要
- セキュリティパッチや機能追加を迅速に全環境に展開する必要がある
- startup.sh等のスクリプト自身も安全に更新できる仕組みが必要

### 1.3 スコープ

本サービスは以下の配布物のバージョン管理を対象とする：
- **コンテナイメージ**: account, terminal, master-data, cart, report, journal, stock, sync（クラウド→エッジ、クラウド→POS）
- **スクリプトファイル**: edge-startup.sh, pos-startup.sh（クラウド→エッジ、クラウド→POS）
- **設定ファイル**: docker-compose.yml, 環境設定ファイル（クラウド→エッジ、クラウド→POS）
- **モジュールファイル**: ホストOS上で実行されるコンポーネント用（クラウド→エッジ、クラウド→POS）
  - Pythonライブラリ（.whl）: ホストOS上のPythonスクリプトが使用するライブラリ（manifest解析、ヘルスチェック等）
  - システムパッケージ（.deb, .rpm）: 監視エージェント、CLIツール等
  - バイナリ（.tar.gz）: Docker外で動作するヘルパーツール
- **画像・ドキュメント**: UIアセット、マニュアル（クラウド→エッジ、クラウド→POS）

## 2. 機能要件

### 2.1 サービス構成

#### 2.1.1 新規機能の追加

Sync Service内に以下の3つの新規機能を追加実装します：

##### バージョン管理API

**機能概要:**
エッジ端末（Edge/POS）のバージョン状態を管理し、更新要否を判定するAPI群。

**主要機能:**
- **エッジ端末認証**: edge_idとsecretによる認証、JWTトークン発行
- **バージョン確認**: デバイスの現在バージョンと目標バージョンを比較し、更新の要否を判定
- **更新履歴管理**: 各デバイスの更新履歴を記録・取得
- **状態管理**: デバイスごとのダウンロード状態、適用状態、スケジュール情報を一元管理

**エンドポイント:**
- POST /api/v1/version-management/auth - 認証・トークン発行
- POST /api/v1/artifact-management/check - バージョン確認・Manifest取得
- GET /api/v1/artifact-management/history - 更新履歴取得

##### Artifact Management API

**機能概要:**
ファイル（スクリプト、設定ファイル、モジュール等）の配布・ダウンロードを管理するAPI群。

**主要機能:**
- **ファイルダウンロード**: Azure Blob Storageからファイルをストリーミング配信
- **ダウンロード完了通知**: Phase 1-3（ダウンロードフェーズ）の完了を受領・記録
- **適用完了通知**: Phase 4-9（適用フェーズ）の完了を受領・記録
- **チェックサム配信**: ファイルの整合性検証に必要なSHA256ハッシュを提供
- **データソースルーティング**: デバイスタイプに応じた最適なダウンロード元を指定

**エンドポイント:**
- POST /api/v1/artifact-management/download - ファイルダウンロード
- POST /api/v1/artifact-management/download-complete - ダウンロード完了通知
- POST /api/v1/artifact-management/apply-complete - 適用完了通知

**特徴:**
- Edge Sync ServiceとCloud Sync Serviceで同一のAPIを提供（ローカル/クラウド透過）
- POS端末はEdge→Cloud自動フォールバック機構により高可用性を実現

##### Manifest Generation

**機能概要:**
デバイスタイプ、店舗、バージョン情報に基づいて、各デバイス向けの更新Manifestを動的に生成する機能。

**主要機能:**
- **デバイス別カスタマイズ**:
  - Edge端末: `data_source = "cloud"` でACR直接取得
  - POS端末: `data_source = "edge"` でEdge Registry優先、ACRフォールバック
- **店舗別URL生成**:
  - POS端末向けに同一店舗内のEdge端末のホスト名（例: edge-tokyo-001）を動的に解決
  - primary_url, fallback_urlを自動設定
- **メタデータ取得**:
  - Azure Blob Storageからファイルメタデータ（チェックサム、サイズ等）を取得
  - Azure Container Registryからイメージダイジェストを取得
- **スケジュール埋め込み**: 管理者が指定した適用スケジュール（scheduled_at, maintenance_window）をManifestに含める
- **バージョン解決**: target_versionに基づいて配布対象ファイル一覧を決定

**生成タイミング:**
- エッジ端末からのバージョン確認リクエスト時（POST /api/v1/artifact-management/check）
- リアルタイムで最新の情報を反映

**Manifest構成要素:**
```json
{
  "manifest_version": "1.0",
  "device_type": "pos",
  "device_id": "edge-A1234-tokyo-002",
  "target_version": "v1.2.3",
  "artifacts": [...],           // ファイル一覧（data_source, URLs含む）
  "container_images": [...],    // イメージ一覧（data_source, registries含む）
  "apply_schedule": {           // 適用スケジュール
    "scheduled_at": "2025-01-18T02:00:00Z",
    "maintenance_window": {...}
  }
}
```

**設計上の重要性:**
- 静的Manifestでは実現できないデバイスごと・店舗ごとのカスタマイズを可能にする
- 管理者の運用負荷を削減（デバイスごとにManifestファイルを手動作成する必要がない）
- Azure Blob/ACRの最新状態を常に反映（チェックサム、イメージダイジェスト等）

#### 2.1.2 サービス配置

```
クラウド環境:
  - Azure Container Registry (ACR): コンテナイメージ管理（複数バージョンのタグを保持）
  - Azure Blob Storage: スクリプト・設定ファイル管理（バージョンフォルダ形式、最低3世代保持）
    例: /artifacts/v1.2.1/, /artifacts/v1.2.2/, /artifacts/v1.2.3/
  - Sync Service: バージョン管理API、Manifest配信、エッジデバイス認証

エッジ環境:
  - Edge Registry (Harbor/Docker Registry): イメージキャッシュ
  - startup.sh: 自動更新スクリプト
  - Services: 各マイクロサービス

POS端末環境:
  - startup.sh: 自動更新スクリプト
  - Services: 各マイクロサービス

注記: POS端末もバージョン管理においてはエッジデバイスとして扱われ、edge_idで認証
```

### 2.2 バージョン管理仕様

#### 2.2.1 バージョン管理対象一覧

| 配布物種別 | 配布方向 | バージョンチェック | 配布単位 | チェック間隔 | 備考 |
|------------|----------|----------|----------|----------------|------|
| container_images | クラウド→エッジ/POS | ✓ | イメージ単位 | 起動時・15分 | Dockerレイヤー差分 |
| scripts | クラウド→エッジ/POS | ✓ | バージョン一括 | 起動時・15分 | 自己更新対応 |
| modules | クラウド→エッジ/POS | ✓ | バージョン一括 | 起動時・15分 | ホストOS用（.whl, .deb等） |
| configs | クラウド→エッジ/POS | ✓ | バージョン一括 | 起動時・15分 | 環境設定ファイル |
| images | クラウド→エッジ/POS | ✓ | バージョン一括 | 起動時・15分 | UIアセット |

**注記:**
- チェック間隔は環境変数で調整可能（デフォルト: 15分）
- **バージョンチェック**: デバイスの現在バージョンとクラウドの目標バージョンを比較
- **配布単位**:
  - **イメージ単位**: 変更されたコンテナイメージのみダウンロード（Dockerレイヤー差分活用）
  - **バージョン一括**: バージョンフォルダ内の全ファイルを一括配布・更新（ファイル単位の差分判定なし）
- **ストレージ構造**: Azure Blob Storageにバージョンフォルダ形式で保存（最低3世代保持）
  - 例: `/artifacts/v1.2.1/`, `/artifacts/v1.2.2/`, `/artifacts/v1.2.3/`
  - 各バージョンフォルダには、そのバージョンで配布される全ファイルを格納

#### 2.2.2 コンテナイメージ（クラウド → エッジ/POS）

**対象イメージ:**
- account, terminal, master-data, cart, report, journal, stock, sync

**バージョン管理方法:**
- **バージョンタグ規則**: `<service-name>:v<major>.<minor>.<patch>`
- **初期同期**: 一括同期（全イメージをpull）
- **定期同期**: 差分同期（変更されたイメージのみpull）
- **手動同期**: 管理画面から任意のタイミングで実行可能

**一括同期タイミング:**
- システム初期セットアップ時
- 管理者による手動実行時

**差分同期タイミング:**
- 起動時チェック（エッジ端末、POS端末）
- 15分ごとの定期チェック（エッジ端末、POS端末）

**データソースルーティング:**

Manifestにはイメージの取得元（data_source）が記載され、デバイスはこれに従ってイメージを取得：

- **Edge端末**: `data_source = "cloud"`
  - ACRから直接取得

- **POS端末**: `data_source = "edge"` または `"cloud"`
  - `data_source = "edge"`: 同一店舗内のEdge Registryから取得（高速）
  - `data_source = "cloud"`: ACRから直接取得
  - Edge Registryが利用不可の場合は自動的にACRへフォールバック

**レジストリの定義:**
- **優先レジストリ（primary_registry）**: 同一店舗内のEdge端末上で稼働するコンテナレジストリ（例: `edge-tokyo-001:5000`）
  - ローカルネットワーク経由で高速ダウンロード
  - HarborまたはDocker Registryを使用
- **フォールバックレジストリ（fallback_registry）**: クラウド上のAzure Container Registry（例: `masakugel.azurecr.io`）
  - 優先レジストリが利用不可の場合に使用
  - インターネット経由でアクセス

**取得フロー（POS端末の場合）:**
1. Manifestの`data_source`を確認
2. `data_source = "edge"`: 優先的にEdge Registryよりイメージ取得を試行
3. 失敗時: フォールバック先としてクラウドのACRよりイメージ取得
4. チェックサム検証後、適用

#### 2.2.3 スクリプトファイル（クラウド → エッジ/POS）

**対象ファイル:**
- edge-startup.sh
- pos-startup.sh

**バージョン管理方法:**
- **バージョン一括配布**: デバイスのバージョンとクラウドの目標バージョンが異なる場合、該当バージョンフォルダ内の全ファイルを配布
- **自己更新**: スクリプト自身を安全に更新
- **復旧時同期**: ネットワーク復旧時に未適用分を適用

**同期タイミング:**
- 起動時チェック（Phase 1で最優先実行）
- 15分ごとの定期チェック（エッジ端末、POS端末）
- ネットワーク障害時はローカルキャッシュを使用
- ネットワーク復旧時に未適用分を自動適用

**データソースルーティング:**

Manifestには各スクリプトファイルの取得元（data_source）が記載され、デバイスはこれに従ってファイルを取得：

- **Edge端末**: `data_source = "cloud"`
  - Cloud Sync Service API経由で取得

- **POS端末**: `data_source = "edge"` または `"cloud"`
  - `data_source = "edge"`: 同一店舗内のEdge端末のSync Service API経由で取得（高速・ローカルネットワーク）
  - `data_source = "cloud"`: Cloud Sync Service API経由で取得
  - Edge Sync Service APIが利用不可の場合は自動的にCloud Sync Service APIへフォールバック

**重要**: 全てのファイル取得はAPI経由で行われます。ファイルシステムやストレージへの直接アクセスは行いません。

**取得フロー（POS端末の場合）:**
1. Manifestの`data_source`を確認
2. `data_source = "edge"`: `primary_url`（Edge Sync Service API）へリクエスト送信
3. 失敗時: `fallback_url`（Cloud Sync Service API）へリクエスト送信
4. チェックサム検証後、適用

#### 2.2.4 その他ファイル（クラウド → エッジ/POS）

**対象ファイル:**
- **モジュール**（ホストOS上で実行されるコンポーネント用）
  - Pythonライブラリ（.whl）: ホストOS上のPythonスクリプトが使用するライブラリ（manifest解析、ヘルスチェック等）
  - システムパッケージ（.deb, .rpm）: 監視エージェント、CLIツール
  - バイナリ（.tar.gz）: Docker外で動作するヘルパーツール
- **設定ファイル**（.conf, .yaml, .json）
- **画像**（.png, .jpg, .svg）
- **ドキュメント**（.pdf, .md）

**注記**: コンテナ内で使用するPythonライブラリやパッケージは、コンテナイメージに含まれるため、modulesの対象外

**同期方法:**
- **バージョン一括配布**: デバイスのバージョンとクラウドの目標バージョンが異なる場合、該当バージョンフォルダ内の全ファイルを配布
- **復旧時同期**: ネットワーク復旧時に未適用分を適用

**同期タイミング:**
- 起動時チェック
- 15分ごとの定期チェック（エッジ端末、POS端末）

**データソースルーティング:**

スクリプトファイルと同様に、Manifestには各ファイルの取得元が記載：

- **Edge端末**: `data_source = "cloud"`
  - Cloud Sync Service API経由で取得

- **POS端末**: `data_source = "edge"` または `"cloud"`
  - `data_source = "edge"`: Edge Sync Service API経由で取得
  - `data_source = "cloud"`: Cloud Sync Service API経由で取得
  - フォールバック機構により高い可用性を実現

**重要**: 全てのファイル取得はAPI経由で行われます。ファイルシステムやストレージへの直接アクセスは行いません。

**Manifest例（POS端末向け）:**
```json
{
  "artifacts": [
    {
      "type": "script",
      "name": "pos-startup.sh",
      "version": "v1.2.3",
      "data_source": "edge",
      "primary_url": "http://edge-tokyo-001:8007/api/v1/artifact-management/download",
      "fallback_url": "https://sync.kugelpos.cloud/api/v1/artifact-management/download",
      "checksum": "sha256:abc123...",
      "destination": "/opt/kugelpos/pos-startup.sh"
    },
    {
      "type": "config",
      "name": "docker-compose.override.yml",
      "version": "v1.2.3",
      "data_source": "edge",
      "primary_url": "http://edge-tokyo-001:8007/api/v1/artifact-management/download",
      "fallback_url": "https://sync.kugelpos.cloud/api/v1/artifact-management/download",
      "checksum": "sha256:def456...",
      "destination": "/opt/kugelpos/docker-compose.override.yml"
    },
    {
      "type": "module",
      "name": "kugelpos_common-1.2.3-py3-none-any.whl",
      "version": "v1.2.3",
      "data_source": "edge",
      "primary_url": "http://edge-tokyo-001:8007/api/v1/artifact-management/download",
      "fallback_url": "https://sync.kugelpos.cloud/api/v1/artifact-management/download",
      "checksum": "sha256:ghi789...",
      "install_command": "pip install {file}"
    }
  ]
}
```

**注記**:
- Edge端末にもSync Serviceが稼働しており、ローカルにキャッシュされたファイルをAPI経由で配信します
- POS端末は常にAPI経由でファイルを取得し、ファイルシステムには直接アクセスしません

#### 2.2.5 更新処理のフェーズとタイミング

更新処理は以下の9つのフェーズに分かれており、各フェーズは順次実行される。フェーズ間で失敗が発生した場合、適切なロールバック処理が実行される。

**フェーズ1: バージョンチェック**
- **実行タイミング**: 起動時、または15分ごと（定期実行）
- **処理内容**:
  - Sync Serviceに現在のバージョン情報を送信
  - Manifestを取得して更新が必要か判定
- **完了条件**: 更新要否が確定
- **失敗時の動作**: エラーログを記録し、現在のバージョンで起動継続

**フェーズ2: ダウンロード**
- **実行タイミング**: バージョンチェックで更新が必要と判定された直後
- **処理内容**:
  - すべての更新対象ファイル（イメージ、スクリプト等）をダウンロード
  - コンテナイメージは`docker pull`でローカルに保存（まだ適用しない）
  - スクリプトや設定ファイルは一時ディレクトリに保存
- **完了条件**: すべてのファイルがローカルに保存完了
- **失敗時の動作**: ダウンロード済みファイルを削除し、現在のバージョンで起動継続

**フェーズ3: 検証**
- **実行タイミング**: ダウンロード完了直後
- **処理内容**:
  - ダウンロードした各ファイルのチェックサム（SHA256）を検証
  - Manifestに記載されたチェックサムと一致することを確認
- **完了条件**: すべてのファイルのチェックサムが一致
- **失敗時の動作**: ダウンロード済みファイルを削除し、現在のバージョンで起動継続

**フェーズ4: バックアップ**
- **実行タイミング**: 検証完了直後
- **処理内容**:
  - 現在の`docker-compose.yml`をバックアップ
  - 現在のスクリプトファイルをバックアップ
  - 現在のバージョン情報を保存
- **完了条件**: すべての必要ファイルがバックアップ完了
- **失敗時の動作**: バックアップ失敗をログに記録し、更新を中止

**フェーズ5: 適用準備**
- **実行タイミング**: バックアップ完了直後
- **処理内容**:
  - `docker-compose.yml`のイメージタグを新バージョンに更新
  - スクリプトファイルを新バージョンで置き換え
  - 設定ファイルを更新
  - **この時点ではまだサービスは停止していない**
- **完了条件**: すべての設定ファイルが更新完了
- **失敗時の動作**: バックアップから設定ファイルを復元

**フェーズ6: サービス停止**
- **実行タイミング**: 適用準備完了直後
- **処理内容**:
  - `docker-compose down`で既存サービスを停止
  - グレースフルシャットダウン（最大30秒待機）
- **完了条件**: すべてのコンテナが停止
- **失敗時の動作**: 強制停止を実行し、次のフェーズへ進む

**フェーズ7: サービス起動**
- **実行タイミング**: サービス停止完了直後
- **処理内容**:
  - 更新された`docker-compose.yml`で`docker-compose up -d`を実行
  - 新バージョンのコンテナイメージでサービスを起動
  - 起動完了を待機（最大120秒）
- **完了条件**: すべてのコンテナが起動（running状態）
- **失敗時の動作**: フェーズ10のロールバック処理へ移行

**フェーズ8: ヘルスチェック**
- **実行タイミング**: サービス起動完了後、30秒待機してから実行
- **処理内容**:
  - 各サービスの`/health`エンドポイントに対してHTTP GETリクエスト
  - すべてのサービスが200 OKを返すことを確認
  - 最大3回、10秒間隔でリトライ
- **完了条件**: すべてのサービスがヘルスチェックに合格
- **失敗時の動作**: フェーズ10のロールバック処理へ移行

**フェーズ9: 完了通知**
- **実行タイミング**: ヘルスチェック成功直後
- **処理内容**:
  - Sync Serviceに更新完了を通知（`POST /artifact-management/complete`）
  - ローカルのバージョン情報ファイルを更新
  - 古いバックアップファイルを削除
- **完了条件**: 通知が成功（またはオフライン時はキューに保存）
- **失敗時の動作**: エラーログを記録するが、更新は成功とみなす

**フェーズ10: 自動ロールバック（エラー時のみ）**
- **実行タイミング**: フェーズ7またはフェーズ8で失敗した場合
- **処理内容**:
  1. 現在のサービスを停止（`docker-compose down`）
  2. バックアップした`docker-compose.yml`を復元
  3. バックアップしたスクリプトを復元
  4. 直前のバージョンでサービスを起動（`docker-compose up -d`）
  5. ヘルスチェックを実行
  6. Sync Serviceに失敗を通知
- **完了条件**: 直前のバージョンでサービスが正常起動
- **失敗時の動作**: 管理者に緊急アラートを送信

**注記**:
- ロールバックは常に「直前のバージョン」に戻す
- 管理者による任意バージョンへのロールバック指示機能は提供しない
- ファイル内容に不備があった場合は、修正版を新しいバージョンとして配信・適用する

**重要な時間差:**
- **ダウンロードと適用の時間差**: フェーズ2（ダウンロード）完了からフェーズ6（サービス停止）まで、サービスは稼働継続
- **サービス停止時間**: フェーズ6開始からフェーズ7完了まで（通常1-3分）
- **検証期間**: フェーズ8のヘルスチェックで新バージョンの動作を確認（30-60秒）

#### 2.2.6 2段階更新の仕組み

更新処理は常に「**ダウンロードフェーズ**」と「**適用フェーズ**」の2つのステップに分離して実行されます。これにより、営業時間中にファイルをダウンロードしておき、Manifestで指定されたメンテナンス時間帯に自動適用される仕組みです。

**2段階更新フロー:**

| フェーズ | 実行内容 | 実行タイミング | サービス停止 | 通知API |
|---------|---------|--------------|-------------|---------|
| **ダウンロード** | Phase 1-3 | 更新検知後、即座に実行 | なし | `/download-complete` |
| **適用** | Phase 4-9 | Manifestの`scheduled_apply_at`に到達時 | あり（1-3分） | `/apply-complete` |

**ダウンロードフェーズ (Phase 1-3):**
- **実行タイミング**: Sync Service (Client)が15分間隔でバージョンチェックし、更新検知後に即座に実行
- **実行フェーズ**: Phase 1（バージョンチェック）→ Phase 2（ダウンロード）→ Phase 3（検証）
- **処理内容**:
  - クラウドSync ServiceからManifestを取得
  - 更新ファイルをすべてダウンロード
  - チェックサムを検証して完全性を確認
  - ダウンロード済みファイルを`/opt/kugelpos/pending-updates/{version}/`に保存
  - Manifestを`/opt/kugelpos/pending-updates/{version}/manifest.json`に保存
  - クラウドSync Serviceに`POST /download-complete`を送信
- **サービスへの影響**: サービスは継続稼働（停止なし）
- **失敗時の動作**: ダウンロード済みファイルを削除し、エラーを記録してクラウドに通知

**適用フェーズ (Phase 4-9):**
- **実行タイミング**: Manifestの`apply_schedule.scheduled_at`で指定された日時に到達時
- **実行フェーズ**: Phase 3（再検証）→ Phase 4（バックアップ）→ Phase 5（適用準備）→ Phase 6-9（サービス停止・起動・完了）
- **処理内容**:
  - `/opt/kugelpos/pending-updates/{version}/`からダウンロード済みファイルを読み込み
  - チェックサムを再検証（ダウンロード後の改ざん防止）
  - 現在の設定をバックアップ
  - サービスを停止して新バージョンを適用
  - ヘルスチェック実施
  - クラウドSync Serviceに`POST /apply-complete`を送信
- **前提条件**: ダウンロードフェーズが正常完了していること
- **サービスへの影響**: Phase 6-8でサービス停止（通常1-3分）
- **失敗時の動作**: 自動ロールバック処理を実行し、クラウドに通知

**ダウンロード済みファイルの管理:**

ダウンロード済みファイルは以下の構造で管理されます：

```
/opt/kugelpos/pending-updates/
  └── v1.2.3/                          # バージョンディレクトリ
      ├── manifest.json                 # Manifest情報
      ├── status.json                   # ダウンロード状態
      ├── images/                       # コンテナイメージ情報
      │   └── image-list.txt           # pullしたイメージのリスト
      └── artifacts/                    # ファイル群
          ├── pos-startup.sh
          ├── docker-compose.override.yml
          └── kugelpos_common-1.2.3.whl
```

**status.json の例:**
```json
{
  "version": "v1.2.3",
  "download_status": "completed",
  "download_started_at": "2025-01-17T14:30:00Z",
  "download_completed_at": "2025-01-17T14:35:00Z",
  "verification_status": "passed",
  "ready_to_apply": true,
  "artifacts_count": 15,
  "total_size_bytes": 524288000
}
```

**Manifestによる適用スケジュール制御:**

Manifestには適用スケジュール情報が含まれ、Sync Service (Client)がこれに従って自動適用します：

```json
{
  "target_version": "v1.2.3",
  "apply_schedule": {
    "scheduled_at": "2025-01-18T02:00:00Z",  // 適用開始日時（ISO 8601形式）
    "maintenance_window": {
      "start": "02:00",                       // メンテナンス開始時刻
      "end": "05:00"                          // メンテナンス終了時刻（この時刻までリトライ許可）
    }
  }
}
```

**設計思想:**
- Manifestがエッジデバイスに配信される時点で、管理者の承認は完了している
- したがって、ダウンロード完了後は`scheduled_at`で指定された日時に自動的に適用を実行
- デバイス側での個別の承認フローは不要

**maintenance_windowの用途:**
1. **リトライ許容時間帯**: `scheduled_at`での適用失敗時、`end`までリトライを継続
2. **遅延実行の許可**: デバイスが`scheduled_at`を過ぎてバージョンチェックした場合でも、`window`内であれば適用実行
3. **適用期限**: `end`を過ぎた場合は適用をスキップし、次回スケジュールまで待機

**動作例:**
```
scheduled_at: 2025-01-18T02:00:00Z
maintenance_window: 02:00-05:00

ケース1: 正常実行
  2:00 → 適用開始 → 2:03 完了

ケース2: リトライ成功
  2:00 → ネットワーク障害で失敗
  2:15 → リトライ成功（window内のため実行）
  2:18 → 完了

ケース3: リトライ期限超過
  2:00 → 失敗
  2:15 → 失敗
  ...
  5:01 → window外のため適用スキップ、次回スケジュールまで待機

ケース4: 遅延チェック
  デバイスが2:30に起動してバージョンチェック
  → scheduled_at過ぎているが、window内（5:00前）のため適用実行
```

**運用例（自動実行フロー）:**

```
15:00  Sync Service (Client)がバージョンチェック
       → 更新あり（v1.2.3）
       → Manifestを取得: scheduled_at = 2025-01-18 02:00

15:01  即座にダウンロード開始
       → /opt/kugelpos/pending-updates/v1.2.3/ に保存
       → POST /download-complete 送信
       → 状態: waiting_for_schedule（scheduled_at まで待機）

15:30  定期チェック（更新なし、待機継続）

...    scheduled_at まで待機

02:00  scheduled_at に到達
       → 自動的に適用フェーズ開始
       → サービス停止・新バージョン起動
       → ヘルスチェック
       → POST /apply-complete 送信
```

**注意事項:**
- ダウンロード済みファイルの有効期限: 7日間（環境変数で調整可能）
- 複数バージョンがダウンロード済みの場合は最新バージョンを適用
- 適用前に再度チェックサム検証を実施（ダウンロード後の改ざん防止）
- Manifestの配信自体が管理者の承認プロセスを経ているため、デバイス側での追加承認は不要

### 2.3 更新状態管理

**管理主体**: クラウド側Sync Serviceが全デバイスのバージョン状態を一元管理

**管理対象:**
- `device_type`: デバイスタイプ（edge/pos）
- `edge_id`: エッジデバイス識別子
  - **命名規則**: `edge-<tenant_id>-<store_code>-<連番>`
  - **例**:
    - Edge端末: `edge-A1234-tokyo-001`
    - POS端末: `edge-A1234-tokyo-002`
  - **一意性**: グローバルに一意（tenant_id、store_code、連番の組み合わせ）
  - **デバイス種別の判定**: `device_type`フィールドで判別（edge/pos）
- `current_version`: 現在のバージョン
- `target_version`: 目標バージョン
- `update_status`: 更新状態（idle/checking/downloading/download_completed/waiting_for_schedule/applying/apply_completed/failed/auto_rollback）
- `download_status`: ダウンロード状態（not_started/in_progress/completed/failed）
- `download_completed_at`: ダウンロード完了日時
- `apply_status`: 適用状態（not_started/scheduled/in_progress/completed/failed/auto_rollback）
- `scheduled_apply_at`: 適用予定日時（Manifestで指定）
- `apply_completed_at`: 適用完了日時
- `pending_version`: ダウンロード済み未適用バージョン
- `last_check_timestamp`: 最終チェック時刻
- `retry_count`: リトライ回数
- `error_message`: エラーメッセージ

**注記**:
- `edge_id`はデータ同期の仕様と整合性を持たせるため、統一の命名規則を使用
- Edge端末とPOS端末の区別は`device_type`で行う

#### 2.3.1 更新ステータス

クラウド側Sync Serviceが、各デバイス・配布物種別ごとに以下を管理：
- `edge_id`: エッジデバイス識別子
- `artifact_type`: 配布物種別（container_image/script/module/config/image）
- `artifact_name`: 配布物名称
- `current_version`: 現在バージョン
- `target_version`: 目標バージョン
- `last_check_timestamp`: 最終チェック時刻
- `update_status`: 更新状態
- `retry_count`: リトライ回数
- `error_message`: エラーメッセージ

#### 2.3.2 更新履歴

クラウド側Sync Serviceが更新実行履歴を記録：
- `edge_id`: エッジデバイス識別子
- 配布物種別・名称
- 更新タイプ（upgrade/downgrade/auto_rollback）
- 更新開始・終了時刻
- 更新データ件数・サイズ
- 成功/失敗状態
- エラー詳細
- リトライ回数
- 処理時間（ミリ秒）

**注記**:
- `auto_rollback`は適用失敗時の自動ロールバックを示す
- ロールバック先は常に直前のバージョン

### 2.4 競合解決

#### 2.4.1 競合検出

- Manifestで指定されたバージョンと現在のバージョンを比較
- チェックサム（SHA256）による変更追跡

#### 2.4.2 解決戦略

**Server Wins (サーバー優先):**
- 唯一の解決方法としてシンプルに実装
- Manifestで指定されたバージョンを常に採用
- 適用対象：全配布物種別
- 更新失敗時は自動ロールバック

### 2.5 エラーハンドリング

#### 2.5.1 リトライ機構

- 指数バックオフによるリトライ
- 最大リトライ回数: 3回（設定可能）
- リトライ間隔: 1秒 → 2秒 → 4秒

#### 2.5.2 自動ロールバック

- **対象**: サービス起動失敗またはヘルスチェック失敗時のみ
- **ロールバック先**: 常に直前のバージョン（任意バージョンへのロールバックは行わない）
- **処理**: バックアップから直前のバージョンに復元し、サービスを再起動
- **通知**: ロールバック完了後、クラウドSync Serviceに失敗とロールバック実施を通知

**運用方針**:
- ファイル内容に不備があった場合は、修正版を新しいバージョンとして配信・適用
- 管理者による手動ロールバック指示機能は提供しない（運用をシンプルに保つ）

#### 2.5.3 フォールバック

- ネットワーク障害時はローカルキャッシュを使用
- キャッシュ有効期限: 7日間
- 重要な更新は必ず永続化

## 3. 非機能要件

### 3.1 パフォーマンス要件

- **更新時間**: エッジ端末10分以内、POS端末5分以内
- **同時接続**: 1,000台の同時アクセスに対応
- **データ圧縮率**: 50%以上（gzip使用）
- **差分更新効率**: 90%以上の帯域削減（変更がない場合）

### 3.2 可用性要件

- **稼働率**: クラウドサービス99.9%以上、ACR 99.95%以上
- **自動復旧**: ネットワーク復旧後30秒以内に更新再開
- **データ保証**: At-least-once delivery

### 3.3 セキュリティ要件

#### 3.3.1 通信セキュリティ
- **通信暗号化**: TLS 1.2以上による暗号化
- **チェックサム検証**: SHA256によるファイル改ざん検出

#### 3.3.2 エッジ端末認証

**エッジ端末の定義:**
- 1店舗に複数台のエッジ端末が存在する構成をサポート
- レジ自身がエッジ機能を内蔵するケース（POS端末）
- 専用エッジサーバーを設置するケース（Edge端末）
- **edge_id命名規則**: `edge-<tenant_id>-<store_code>-<連番>`
  - 例: `edge-A1234-tokyo-001` (Edge端末), `edge-A1234-tokyo-002` (POS端末)
  - 一意性: グローバルに一意（tenant_id、store_code、連番の組み合わせ）
  - デバイス種別: `device_type`フィールドで判別（edge/pos）

**認証フロー:**
1. エッジ端末が`edge_id`と`secret`を使用して認証リクエスト送信
2. クラウド側Sync Serviceが認証し、JWTトークンを発行
3. JWTトークンに`tenant_id`、`store_code`、`edge_id`、`device_type`を含める
4. 以降のAPIアクセスはJWTトークンをAuthorizationヘッダーに付与

**JWT構成:**
```json
{
  "tenant_id": "A1234",
  "store_code": "tokyo",
  "edge_id": "edge-A1234-tokyo-001",
  "device_type": "pos",     // pos/edge等
  "exp": 1234567890,        // 有効期限
  "iat": 1234567800         // 発行時刻
}
```

**エッジ端末管理:**
- 各テナントのDB（`sync_{tenant_id}`）にエッジ端末情報を管理
- edge_idはグローバルに一意（tenant_id、store_code、連番の組み合わせ）
- テナント間の完全な分離を実現

**注記:**
- POS端末は業務処理においては`terminal_id`と`api_key`でTerminal Service経由で認証
- バージョン管理においては`edge_id`でSync Serviceに直接認証
- これにより認証フローがシンプルになり、Terminal Serviceへの依存を削減

### 3.4 運用要件

- **監視**: メトリクス、ログ、ヘルスチェック
- **アラート**: 更新失敗、エラー率上昇時に通知
- **バックアップ**: バージョン設定の定期バックアップ

#### 3.4.1 2段階更新の運用ガイドライン

**自動実行の仕組み:**

更新処理は**Sync Service (Client)が自動的に実行**します。管理者はManifestで適用スケジュールを指定するだけです。

1. **ダウンロードフェーズ**: 更新検知後、即座に自動実行（サービス停止なし）
2. **適用フェーズ**: Manifestの`scheduled_apply_at`で指定した日時に自動実行（サービス停止あり）

**推奨運用パターン:**

1. **POS端末向けManifest設定:**
   ```json
   {
     "apply_schedule": {
       "scheduled_at": "2025-01-18T02:00:00Z",  // 深夜2:00に適用
       "maintenance_window": {
         "start": "02:00",
         "end": "05:00"
       }
     }
   }
   ```
   - **ダウンロード**: 更新検知後、営業時間中でも即座に実行（影響なし）
   - **適用**: 営業終了後のメンテナンス時間帯（02:00-05:00）に自動適用
   - **確認**: 営業開始前にヘルスチェック結果を確認

2. **Edge端末向けManifest設定:**
   ```json
   {
     "apply_schedule": {
       "scheduled_at": "2025-01-18T01:00:00Z",  // 深夜1:00に適用
       "maintenance_window": {
         "start": "01:00",
         "end": "04:00"
       }
     }
   }
   ```
   - **ダウンロード**: 更新検知後、業務時間中でも即座に実行
   - **適用**: 深夜のメンテナンスウィンドウ（01:00-04:00）に自動適用
   - **確認**: 適用後、店舗内POS端末への影響を確認

**監視項目:**

- ダウンロード完了状態の監視（クラウドSync Serviceのダッシュボード）
- ダウンロード済みファイルの有効期限チェック（7日間）
- 適用待ちデバイスの一覧確認
- 適用失敗時のアラート通知
- `download_status`、`apply_status`の状態監視

**注意事項:**

- Sync Service (Client)は常にバックグラウンドで稼働し、15分間隔でバージョンチェックを実行
- ダウンロードは更新検知後、自動的に即座に開始
- 適用は`scheduled_apply_at`の日時に到達すると自動的に開始
- 複数バージョンがダウンロード済みの場合、最新バージョンを優先して適用

## 4. インターフェース仕様

### 4.1 エッジ端末認証API

**目的**: エッジ端末（Edge/POS）がクラウドSync Serviceに対して認証を行い、JWTトークンを取得する。以降のバージョン管理APIアクセスには、このトークンが必要となる。

**エンドポイント**: POST /api/v1/version-management/auth

**リクエストヘッダー**:
- Content-Type: application/json

**リクエストパラメータ**:
- tenant_id: テナントID
- store_code: 店舗コード
- edge_id: エッジ端末ID（`edge-<tenant_id>-<store_code>-<連番>`形式、グローバルに一意）
- device_type: デバイスタイプ（edge/pos）
- secret: 認証用シークレット（デバイス初期セットアップ時にクラウド側で発行、SHA256ハッシュ化して保存）

**成功レスポンス（200 OK）**:
- success: true
- data:
  - access_token: JWTアクセストークン
  - token_type: トークンタイプ（bearer）
  - expires_in: 有効期限（秒、デフォルト3600秒）
  - tenant_id: テナントID
  - store_code: 店舗コード
  - edge_id: エッジ端末ID
  - device_type: デバイスタイプ（edge/pos）

**エラーレスポンス（401 Unauthorized）**:
- success: false
- error:
  - code: エラーコード（AUTH_002: 無効なedge_id等）
  - message: エラーメッセージ

### 4.2 バージョン管理API

#### 4.2.1 バージョン確認（エッジ端末からの定期リクエスト）

**目的**: エッジ端末が定期的に（起動時、15分ごと）クラウドSync Serviceに対して現在のバージョン情報を送信し、更新が必要かどうかを確認する。更新が必要な場合は、ダウンロード対象のファイル一覧とManifest情報が返却される。

**エンドポイント**: POST /api/v1/artifact-management/check

**リクエストヘッダー**:
- Authorization: Bearer <JWT_TOKEN>
- Content-Type: application/json

**リクエストパラメータ**:
- device_type: デバイスタイプ（edge/pos）
- edge_id: エッジ端末ID（例: edge-A1234-tokyo-001）
- current_version: 現在適用中のバージョン（例: v1.2.2）
- current_checksums（オプション）: 現在のファイルチェックサム情報
  - キー: ファイル名またはサービス名
  - 値: SHA256チェックサム文字列

**成功レスポンス（200 OK）**:
- success: true
- data:
  - updates_required: 更新が必要かどうか（真偽値）
  - target_version: 目標バージョン（例: v1.2.3）
  - artifacts: ダウンロード対象のファイル一覧（配列）
    - type: ファイル種別（script/module/config/image/doc）
    - name: ファイル名
    - version: ファイルのバージョン
    - data_source: データ取得元（cloud/edge）
      - Edge端末の場合: cloud（クラウドSync Serviceから取得）
      - POS端末の場合: edge（店舗内Edge端末から取得）またはcloud
    - primary_url: 優先ダウンロードURL
    - fallback_url: フォールバックURL（Edge端末の場合はnull、POS端末の場合はクラウドURL）
    - checksum: SHA256チェックサム
    - size: ファイルサイズ（バイト）
    - destination: 配置先パス
    - permissions: ファイル権限（8進数文字列、例: 0755）
    - priority: 適用優先度（数値、小さいほど優先）
    - required: 必須ファイルかどうか（真偽値）
  - container_images: ダウンロード対象のコンテナイメージ一覧（配列）
    - service: サービス名（例: cart）
    - version: イメージバージョン
    - data_source: データ取得元（cloud/edge）
    - primary_registry: 優先レジストリ（Edge端末の場合はACR、POS端末の場合は店舗内Edge Registry）
    - primary_image: 優先レジストリでのイメージ名
    - fallback_registry: フォールバックレジストリ（クラウドACR）
    - fallback_image: フォールバックレジストリでのイメージ名
    - checksum: SHA256チェックサム
  - apply_schedule: 適用スケジュール情報
    - scheduled_at: 適用予定日時（ISO 8601形式）
    - maintenance_window: メンテナンスウィンドウ
      - start: 開始時刻（HH:MM形式）
      - end: 終了時刻（HH:MM形式）

#### 4.2.2 更新履歴取得

**目的**: 指定されたエッジ端末の更新履歴を取得する。期間を指定して履歴を絞り込むことができる。

**エンドポイント**: GET /api/v1/artifact-management/history

**リクエストヘッダー**:
- Authorization: Bearer <JWT_TOKEN>

**クエリパラメータ**:
- edge_id: 履歴を取得するエッジ端末ID
- from（オプション）: 開始日付（YYYY-MM-DD形式）
- to（オプション）: 終了日付（YYYY-MM-DD形式）

**成功レスポンス（200 OK）**:
- success: true
- data:
  - histories: 更新履歴配列
    - update_id: 更新ID（一意な識別子）
    - edge_id: エッジ端末ID
    - from_version: 更新前のバージョン
    - to_version: 更新後のバージョン
    - update_type: 更新種別（upgrade/downgrade/auto_rollback）
    - start_time: 更新開始時刻（ISO 8601形式）
    - end_time: 更新終了時刻（ISO 8601形式）
    - status: 更新結果（success/failed）
    - error_message: エラーメッセージ（失敗時のみ）
  - total: 取得された履歴の総数

### 4.3 Artifact配布API

#### 4.3.1 ファイルダウンロード（エッジ端末からクラウドへ）

**目的**: エッジ端末がSync Service APIからファイル（スクリプト、設定ファイル、モジュール等）をダウンロードする。バイナリデータとして返却される。

**エンドポイント**: POST /api/v1/artifact-management/download

**リクエストヘッダー**:
- Authorization: Bearer <JWT_TOKEN>
- Content-Type: application/json

**リクエストパラメータ**:
- artifact_type: ファイル種別（script/module/config/image/doc）
- artifact_name: ファイル名（例: edge-startup.sh）
- version: 要求するバージョン（例: v1.2.3）
- checksum: 期待するSHA256チェックサム

**成功レスポンス（200 OK）**:
- レスポンスヘッダー:
  - Content-Type: application/octet-stream
  - Content-Disposition: attachment; filename="ファイル名"
  - X-Checksum: SHA256チェックサム
  - X-File-Size: ファイルサイズ（バイト）
- レスポンスボディ: ファイルのバイナリデータ

**注記**: エッジ端末はダウンロード後、X-Checksumヘッダーの値とダウンロードしたファイルのチェックサムを照合し、一致することを確認する必要がある。

#### 4.3.2 ダウンロード完了通知

**目的**: ダウンロードフェーズ（Phase 1-3）完了時にエッジ端末からクラウドSync Serviceへ通知する。ダウンロードの成功・失敗状況、検証結果をクラウド側で記録し、適用待ち状態として管理する。

**エンドポイント**: POST /api/v1/artifact-management/download-complete

**リクエストヘッダー**:
- Authorization: Bearer <JWT_TOKEN>
- Content-Type: application/json

**リクエストパラメータ**:
- device_type: デバイスタイプ（edge/pos）
- edge_id: エッジ端末ID
- target_version: ダウンロード対象バージョン（例: v1.2.3）
- download_status: ダウンロード結果（completed/failed）
- download_started_at: ダウンロード開始時刻（ISO 8601形式）
- download_completed_at: ダウンロード完了時刻（ISO 8601形式）
- artifacts_count（オプション）: ダウンロードしたファイル数
- total_size_bytes（オプション）: ダウンロード総サイズ（バイト）
- verification_status（オプション）: チェックサム検証結果（passed/failed）
- pending_updates_path（オプション）: ダウンロード済みファイルの保存先パス
- error_message（オプション）: エラーメッセージ（失敗時のみ）
- timestamp: 通知送信時刻（ISO 8601形式）

**成功レスポンス（200 OK）**:
- success: true
- data:
  - status: download_acknowledged（ダウンロード完了を受領）
  - message: 確認メッセージ
  - pending_version: ダウンロード済み未適用バージョン
  - scheduled_apply_at: Manifestで指定された適用予定日時（ISO 8601形式）

**注記**: ダウンロードが失敗した場合、download_statusをfailedに設定し、error_messageにエラー内容を記載して送信する。チェックサム検証失敗などの詳細情報を含めることで、クラウド側での問題分析が可能になる。

#### 4.3.3 適用完了通知

**目的**: 適用フェーズ（Phase 4-9）完了時にエッジ端末からクラウドSync Serviceへ通知する。適用の成功・失敗・ロールバック状況、サービス停止時間、ヘルスチェック結果をクラウド側で記録する。

**エンドポイント**: POST /api/v1/artifact-management/apply-complete

**リクエストヘッダー**:
- Authorization: Bearer <JWT_TOKEN>
- Content-Type: application/json

**リクエストパラメータ**:
- device_type: デバイスタイプ（edge/pos）
- edge_id: エッジ端末ID
- target_version: 適用対象バージョン（例: v1.2.3）
- apply_status: 適用結果（success/failed/auto_rollback）
- apply_started_at: 適用開始時刻（ISO 8601形式）
- apply_completed_at: 適用完了時刻（ISO 8601形式）
- scheduled_apply_at: Manifestで指定された適用予定日時（ISO 8601形式）
- downtime_seconds（オプション）: サービス停止時間（秒）
- health_check_status（オプション）: ヘルスチェック結果（passed/failed）
- update_results（オプション）: 個別ファイル・イメージの更新結果（配列）
  - name または service: ファイル名またはサービス名
  - type: 種別（script/container_image/module/config等）
  - version: 適用されたバージョン
  - status: 適用結果（success/failed）
  - error: エラーメッセージ（失敗時のみ）
- rolled_back_to_version（オプション）: 自動ロールバック先バージョン（常に直前のバージョン、auto_rollback時のみ）
- error_message（オプション）: エラーメッセージ（失敗時のみ）
- timestamp: 通知送信時刻（ISO 8601形式）

**成功レスポンス（200 OK）**:
- success: true
- data:
  - status: apply_acknowledged（適用完了を受領）
  - message: 確認メッセージ
  - current_version: 現在適用されているバージョン
  - next_check: 次回バージョンチェック予定日時（ISO 8601形式）

**注記**:
- 適用が失敗して自動ロールバックが実行された場合、apply_statusを`auto_rollback`に設定し、rolled_back_to_versionに直前のバージョン、error_messageにエラー内容を記載して送信する
- ヘルスチェック失敗などの詳細情報を含めることで、クラウド側での問題分析が可能になる
- ファイル内容に不備があった場合は、修正版を新しいバージョンとして配信・適用する運用とする

## 5. 配布物詳細仕様

### 5.1 ファイルカテゴリ

| カテゴリ | type値 | 拡張子例 | 用途 |
|---------|--------|---------|------|
| スクリプト | script | .sh, .bash | 起動・更新処理 |
| モジュール | module | .whl, .deb, .rpm, .tar.gz | プログラム機能追加 |
| 画像 | image | .png, .jpg, .svg | UI表示 |
| 設定 | config | .conf, .yaml, .json | サービス設定 |
| ドキュメント | doc | .pdf, .md | マニュアル |

### 5.2 各カテゴリの処理要件

#### スクリプトファイル
- バックアップを自動作成（前バージョンのみ保持）
- 実行権限を自動設定（755）
- ロールバック時は前バージョンに復元

#### モジュール（ホストOS用）
- **対象**: ホストOS上で実行されるコンポーネントの依存関係
  - Pythonライブラリ: ホストOS上のPythonスクリプトが使用（manifest解析、ヘルスチェック等）
  - 監視エージェント、CLIツール等
- **インストール方法**:
  - Python: `pip install {file}` で自動インストール
  - システムパッケージ: `apt install {file}` または `yum install {file}` で自動インストール
  - バイナリ: 指定ディレクトリに展開
- **ロールバック**: パッケージマネージャーの機能を利用（`pip uninstall`, `apt remove`等）
- **注記**: コンテナ内のライブラリはコンテナイメージに含まれるため対象外

#### 画像ファイル
- 指定パスにコピー
- 権限は644
- ロールバック時は前バージョンに復元

#### 設定ファイル
- 展開処理（tar.gz等）をサポート
- 環境変数の置換（オプション）
- ロールバック時は前バージョンに復元

#### ドキュメント
- 指定パスにコピー
- 圧縮形式もサポート
- ロールバック時は前バージョンに復元

## 6. データベース構成

### 6.1 テナント別データベース

- バージョン管理サービスも他のサービスと同様に、テナント毎に独立したデータベースを使用
- データベース名: `sync_{tenant_id}` (例: sync_A1234)
- 各テナントのバージョン管理状態、履歴、デバイス情報は完全に分離して管理
