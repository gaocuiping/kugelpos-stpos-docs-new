# アプリケーションバージョン管理システム - 機能要件書

## 1. 概要

### 1.1 目的

Kugelposシステムにおけるコンテナイメージ、スクリプト、設定ファイル等の配布物を一元管理し、クラウド・エッジ・POS端末間で効率的かつセキュアにバージョン管理・更新を実現する。

### 1.2 背景

- クラウド、エッジ、POS端末で動作する複数のマイクロサービスのバージョンを統一的に管理する必要がある
- ネットワーク障害時でもエッジ・POS端末が動作を継続できるよう、オフライン耐性が必要
- セキュリティパッチや機能追加を迅速に全環境に展開する必要がある
- startup.sh等のスクリプト自身も安全に更新できる仕組みが必要

### 1.3 スコープ

本サービスは以下の配布物のバージョン管理を対象とする：
- **コンテナイメージ**: account, terminal, master-data, cart, report, journal, stock, sync（クラウド→エッジ、クラウド→POS）
- **スクリプトファイル**: edge-startup.sh, pos-startup.sh（クラウド→エッジ、クラウド→POS）
- **設定ファイル**: docker-compose.yml, 環境設定ファイル（クラウド→エッジ、クラウド→POS）
- **モジュールファイル**: Python wheels, システムパッケージ（クラウド→エッジ、クラウド→POS）
- **画像・ドキュメント**: UIアセット、マニュアル（クラウド→エッジ、クラウド→POS）

## 2. 機能要件

### 2.1 サービス構成

#### 2.1.1 新規機能の追加

- **バージョン管理API**: Sync Service内に実装
- **Artifact Management API**: ファイル配布・ダウンロード機能
- **Manifest Generation**: デバイス別更新内容の生成

#### 2.1.2 サービス配置

```
クラウド環境:
  - Azure Container Registry (ACR): コンテナイメージ管理
  - Azure Blob Storage: スクリプト・設定ファイル管理
  - Sync Service: バージョン管理API、Manifest配信
  - Terminal Service: POS端末認証

エッジ環境:
  - Edge Registry (Harbor/Docker Registry): イメージキャッシュ
  - startup.sh: 自動更新スクリプト
  - Services: 各マイクロサービス

POS端末環境:
  - startup.sh: 自動更新スクリプト
  - Services: 各マイクロサービス
```

### 2.2 バージョン管理仕様

#### 2.2.1 バージョン管理対象一覧

| 配布物種別 | 配布方向 | 差分更新 | 一括更新 | チェック間隔 | 備考 |
|------------|----------|----------|----------|----------------|------|
| container_images | クラウド→エッジ/POS | ✓ | ✓ | 起動時・6時間 | Dockerレイヤー差分 |
| scripts | クラウド→エッジ/POS | ✓ | × | 起動時・6時間 | 自己更新対応 |
| modules | クラウド→エッジ/POS | ✓ | × | 起動時・6時間 | pip, apt等で自動インストール |
| configs | クラウド→エッジ/POS | ✓ | × | 起動時・6時間 | 環境設定ファイル |
| images | クラウド→エッジ/POS | ✓ | × | 起動時・6時間 | UIアセット |

**注記:**
- ✓: サポート、×: 非サポート
- チェック間隔は環境変数で調整可能

#### 2.2.2 コンテナイメージ（クラウド → エッジ/POS）

**対象イメージ:**
- account, terminal, master-data, cart, report, journal, stock, sync

**バージョン管理方法:**
- **バージョンタグ規則**: `<service-name>:v<major>.<minor>.<patch>`
- **初期同期**: 一括同期（全イメージをpull）
- **定期同期**: 差分同期（変更されたイメージのみpull）
- **手動同期**: 管理画面から任意のタイミングで実行可能

**一括同期タイミング:**
- システム初期セットアップ時
- 管理者による手動実行時

**差分同期タイミング:**
- 起動時チェック（エッジ端末、POS端末）
- 6時間ごとの定期チェック（POS端末）

**イメージ取得優先順位（POS端末）:**
1. **エッジレジストリから取得**（高速）
2. **ACRから直接取得**（フォールバック）

#### 2.2.3 スクリプトファイル（クラウド → エッジ/POS）

**対象ファイル:**
- edge-startup.sh
- pos-startup.sh

**バージョン管理方法:**
- **差分同期**: チェックサムベースの差分検出
- **自己更新**: スクリプト自身を安全に更新
- **復旧時同期**: ネットワーク復旧時に未適用分を適用

**同期タイミング:**
- 起動時チェック（Phase 1で最優先実行）
- 6時間ごとの定期チェック（POS端末）
- ネットワーク障害時はローカルキャッシュを使用
- ネットワーク復旧時に未適用分を自動適用

#### 2.2.4 その他ファイル（クラウド → エッジ/POS）

**対象ファイル:**
- モジュール（.whl, .deb, .rpm, .tar.gz）
- 設定ファイル（.conf, .yaml, .json）
- 画像（.png, .jpg, .svg）
- ドキュメント（.pdf, .md）

**同期方法:**
- **差分同期**: チェックサムベースの差分検出
- **復旧時同期**: ネットワーク復旧時に未適用分を適用

**同期タイミング:**
- 起動時チェック
- 6時間ごとの定期チェック（POS端末）

### 2.3 更新状態管理

**管理主体**: クラウド側Sync Serviceが全デバイスのバージョン状態を一元管理

**管理対象:**
- `device_type`: デバイスタイプ（edge/pos）
- `device_id`: デバイス識別子
- `current_version`: 現在のバージョン
- `target_version`: 目標バージョン
- `update_status`: 更新状態（idle/checking/downloading/applying/completed/failed）
- `last_check_timestamp`: 最終チェック時刻
- `last_update_timestamp`: 最終更新時刻
- `retry_count`: リトライ回数
- `error_message`: エラーメッセージ

#### 2.3.1 更新ステータス

クラウド側Sync Serviceが、各デバイス・配布物種別ごとに以下を管理：
- `device_id`: デバイス識別子
- `artifact_type`: 配布物種別（container_image/script/module/config/image）
- `artifact_name`: 配布物名称
- `current_version`: 現在バージョン
- `target_version`: 目標バージョン
- `last_check_timestamp`: 最終チェック時刻
- `update_status`: 更新状態
- `retry_count`: リトライ回数
- `error_message`: エラーメッセージ

#### 2.3.2 更新履歴

クラウド側Sync Serviceが更新実行履歴を記録：
- デバイスID
- 配布物種別・名称
- 更新タイプ（upgrade/downgrade/rollback）
- 更新開始・終了時刻
- 更新データ件数・サイズ
- 成功/失敗状態
- エラー詳細
- リトライ回数
- 処理時間（ミリ秒）

### 2.4 競合解決

#### 2.4.1 競合検出

- Manifestで指定されたバージョンと現在のバージョンを比較
- チェックサム（SHA256）による変更追跡

#### 2.4.2 解決戦略

**Server Wins (サーバー優先):**
- 唯一の解決方法としてシンプルに実装
- Manifestで指定されたバージョンを常に採用
- 適用対象：全配布物種別
- 更新失敗時は自動ロールバック

### 2.5 エラーハンドリング

#### 2.5.1 リトライ機構

- 指数バックオフによるリトライ
- 最大リトライ回数: 3回（設定可能）
- リトライ間隔: 1秒 → 2秒 → 4秒

#### 2.5.2 自動ロールバック

- ヘルスチェック失敗時の自動ロールバック
- チェックサム検証失敗時の自動ロールバック
- 前バージョンへの即座の復帰

#### 2.5.3 フォールバック

- ネットワーク障害時はローカルキャッシュを使用
- キャッシュ有効期限: 7日間
- 重要な更新は必ず永続化

## 3. 非機能要件

### 3.1 パフォーマンス要件

- **更新時間**: エッジ端末10分以内、POS端末5分以内
- **同時接続**: 1,000台の同時アクセスに対応
- **データ圧縮率**: 50%以上（gzip使用）
- **差分更新効率**: 90%以上の帯域削減（変更がない場合）

### 3.2 可用性要件

- **稼働率**: クラウドサービス99.9%以上、ACR 99.95%以上
- **自動復旧**: ネットワーク復旧後30秒以内に更新再開
- **データ保証**: At-least-once delivery

### 3.3 セキュリティ要件

#### 3.3.1 通信セキュリティ
- **通信暗号化**: TLS 1.3による暗号化
- **チェックサム検証**: SHA256によるファイル改ざん検出

#### 3.3.2 エッジ端末認証

**認証フロー:**
1. エッジ端末が`edge_id`を使用して認証リクエスト送信
2. クラウド側Sync Serviceが認証し、JWTトークンを発行
3. 以降のAPIアクセスはJWTトークンをAuthorizationヘッダーに付与

#### 3.3.3 POS端末認証

**認証フロー:**
1. POS端末が`terminal_id`と`api_key`を使用して認証リクエスト送信
2. クラウド側Sync ServiceがTerminal Serviceに検証依頼
3. Terminal Serviceが端末IDとAPIキーを検証
4. 認証成功時のみManifestを返却

### 3.4 運用要件

- **監視**: メトリクス、ログ、ヘルスチェック
- **アラート**: 更新失敗、エラー率上昇時に通知
- **バックアップ**: バージョン設定の定期バックアップ

## 5. 配布物詳細仕様

### 5.1 ファイルカテゴリ

| カテゴリ | type値 | 拡張子例 | 用途 |
|---------|--------|---------|------|
| スクリプト | script | .sh, .bash | 起動・更新処理 |
| モジュール | module | .whl, .deb, .rpm, .tar.gz | プログラム機能追加 |
| 画像 | image | .png, .jpg, .svg | UI表示 |
| 設定 | config | .conf, .yaml, .json | サービス設定 |
| ドキュメント | doc | .pdf, .md | マニュアル |

### 5.2 各カテゴリの処理要件

#### スクリプトファイル
- バックアップを自動作成（前バージョンのみ保持）
- 実行権限を自動設定（755）
- ロールバック時は前バージョンに復元

#### モジュール
- Python: pip install コマンドで自動インストール
- システムパッケージ: apt/yum で自動インストール
- ロールバック: パッケージマネージャーの機能を利用

#### 画像ファイル
- 指定パスにコピー
- 権限は644
- ロールバック時は前バージョンに復元

#### 設定ファイル
- 展開処理（tar.gz等）をサポート
- 環境変数の置換（オプション）
- ロールバック時は前バージョンに復元

#### ドキュメント
- 指定パスにコピー
- 圧縮形式もサポート
- ロールバック時は前バージョンに復元

## 6. データベース構成

### 6.1 テナント別データベース

- バージョン管理サービスも他のサービスと同様に、テナント毎に独立したデータベースを使用
- データベース名: `sync_{tenant_id}` (例: sync_A1234)
- 各テナントのバージョン管理状態、履歴、デバイス情報は完全に分離して管理
