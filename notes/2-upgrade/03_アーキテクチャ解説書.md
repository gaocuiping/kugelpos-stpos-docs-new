# アプリケーションバージョン管理システム - アーキテクチャ解説書

## 1. システム全体構成

### 1.1 システム概要

Kugelpos POSシステムにおけるアプリケーションバージョン管理システムは、クラウド環境からエッジ端末およびPOS端末へのアプリケーション更新を安全かつ効率的に配信する中核的なシステムです。Azure Container Registry (ACR)とAzure Blob Storageを活用し、Manifestベースのバージョン管理により、コンテナイメージとスクリプトファイルの自動更新を実現します。

### 1.3 システムの特徴

- **Manifest駆動**: すべての更新はManifestファイルによって管理
- **データソースルーティング**: Manifestベースの取得元指定により、Edge端末はACR直接、POS端末はEdge Registry優先で取得
- **差分更新**: チェックサムベースの差分検出により効率的な更新
- **自動ロールバック**: ヘルスチェック失敗時の自動復旧
- **自己更新**: スクリプト自身の自動更新機能
- **階層型配布**: Edge Registry経由のPOS端末への高速配布
- **フォールバック機構**: Edge Registry障害時のACR直接アクセス
- **オフライン対応**: ローカルキャッシュによる業務継続性

## 2. コンポーネントアーキテクチャ

### 2.1 主要コンポーネント

#### クラウド側

| コンポーネント | 役割 | 技術スタック |
|--------------|------|------------|
| **Azure Container Registry** | 全バージョンのコンテナイメージを集中管理 | ACR Premium (Geo-replication) |
| **Azure Blob Storage** | スクリプト、設定、画像ファイル等の配布物を集中管理 | Azure Blob Storage (Premium) |

**注記**: POS端末は業務処理においてはTerminal Serviceで認証されるが、バージョン管理ではedge_idでSync Serviceに直接認証

#### エッジ側

| コンポーネント | 役割 | 技術スタック |
|--------------|------|------------|
| **Sync Service (Edge)** | ローカルキャッシュされたアーティファクトをAPI経由で配信、同一店舗内POS端末へのファイル配布 | FastAPI, ファイルシステム |
| **Edge Registry** | コンテナイメージのローカルキャッシュ | Harbor/Docker Registry/Nexus |

| **Local Cache** | Manifest、Artifactキャッシュ | ファイルシステム |

#### POS端末側

| コンポーネント | 役割 | 技術スタック |
|--------------|------|------------|
| **pos-startup.sh** | 起動時更新チェック、自動更新 | Bash Script, systemd |
| **POS Services** | ローカル動作するサービス群 | Docker/Docker Compose |
| **POS Application** | POSフロントエンドアプリ | - |

### 4.4 データソースルーティング詳細

データソースルーティングは、Manifestに含まれる`data_source`フィールドに基づいて、デバイスがどこからコンテナイメージを取得するかを制御する仕組みです。

#### 4.4.1 ルーティング戦略

| デバイスタイプ | data_source | primary_registry | fallback_registry | 理由 |
|-------------|-------------|-----------------|-------------------|------|
| **Edge端末** | `"cloud"` | ACR | - | Edge端末は初回取得元として常にクラウドACRを使用 |
| **POS端末** | `"edge"` | Edge Registry | ACR | 同一店舗内のEdge Registryから高速取得、障害時はACRへフォールバック |

#### 4.4.3 ルーティング決定ロジック

Sync Serviceは以下の情報に基づいてデータソースを決定します:

1. **デバイスタイプ判定**: JWT Tokenから`device_type`を取得
2. **店舗情報取得**: `edge_id`から`store_code`を抽出
3. **Edge端末情報検索**: 店舗コードに対応するEdge端末のホスト名を取得
4. **Manifest生成**:

   **コンテナイメージ:**
   - Edge端末: ACRを`primary_registry`として設定
   - POS端末: Edge Registryを`primary_registry`、ACRを`fallback_registry`として設定

   **アーティファクト（スクリプト、設定ファイル等）:**
   - Edge端末: Cloud Sync Service APIのURLを`primary_url`として設定
   - POS端末: Edge Sync Service APIのURLを`primary_url`、Cloud Sync Service APIのURLを`fallback_url`として設定
   - **重要**: 全てAPI経由で取得（ファイルシステムやストレージへの直接アクセスは行わない）

#### 4.4.4 利点

**コンテナイメージ:**
- **帯域削減**: 同一店舗内のPOS端末は高速なローカルネットワークからイメージ取得
- **レイテンシ削減**: Edge Registryからの取得は通常10-100倍高速
- **耐障害性**: Edge Registry障害時でもACRから取得可能
- **コスト最適化**: ACRからの送信トラフィックを削減

**アーティファクト（スクリプト、設定ファイル等）:**
- **高速配布**: Edge Sync Service API経由でローカルキャッシュから取得（ローカルネットワーク）
- **クラウド負荷軽減**: Cloud Sync Service経由のダウンロード数を削減
- **キャッシュ効果**: Edge端末が一度ダウンロードすれば、店舗内の全POS端末で共有
- **API統一**: 全てのファイル取得がAPI経由で行われるため、セキュリティとアクセス制御が一元管理される
- **フォールバック機構**: Edge Sync API障害時でもCloud Sync API経由で取得可能

**共通:**
- **統一アーキテクチャ**: コンテナイメージとアーティファクトで同じルーティング戦略
- **運用効率**: 1つの仕組みで全ての配布物を管理
- **セキュリティ**: API経由のアクセスにより、認証・認可・監査ログが統一的に管理される

### 5.1 エッジ端末の起動・更新シーケンス

```mermaid
sequenceDiagram
    participant ES as edge-startup.sh
    participant SYNC as Sync Service<br/>(Cloud)
    participant ACR as Azure Container<br/>Registry
    participant DC as Docker Compose
    participant ER as Edge Registry

    Note over ES: システム起動

    ES->>ES: 設定ファイル読込
    ES->>ES: edge_id取得

    ES->>SYNC: POST /version-management/auth
    SYNC->>SYNC: edge_id検証
    SYNC-->>ES: JWT Token発行

    ES->>SYNC: POST /artifact-management/check<br/>(with JWT Token)
    SYNC->>SYNC: バージョン設定確認
    SYNC->>SYNC: Manifest生成<br/>(data_source = "cloud")
    SYNC-->>ES: Manifest応答<br/>(container_images[].data_source = "cloud")

    alt 更新が必要
        loop 各サービス
            ES->>ES: Manifestから<br/>primary_registry確認
```

### 5.4 更新処理の詳細フェーズ（ダウンロードと適用のタイミング）

更新処理は9つのフェーズに分かれ、**ダウンロード（フェーズ2）から実際の適用（フェーズ6）まで、サービスは稼働を継続**します。

```mermaid
sequenceDiagram
    participant DEVICE as Edge/POS Device
    participant SYNC as Sync Service
    participant ACR as Container Registry
    participant DOCKER as Docker Engine
    participant SERVICE as Running Services

    Note over DEVICE,SERVICE: ===== サービス稼働中 =====

    rect rgb(220, 240, 255)
        Note over DEVICE: フェーズ1: バージョンチェック
        DEVICE->>SYNC: POST /artifact-management/check
        SYNC-->>DEVICE: Manifest応答
        DEVICE->>DEVICE: 更新必要性を判定
    end

    rect rgb(255, 240, 220)
        Note over DEVICE: フェーズ2: ダウンロード
        Note over DEVICE: ⚠️ サービスは稼働継続
        loop 各イメージ
            DEVICE->>ACR: docker pull (一時保存)
            ACR-->>DEVICE: イメージダウンロード
            DEVICE->>DOCKER: ローカルに保存
            Note right of DOCKER: まだ適用しない
        end
        Note over DEVICE: すべてのファイルを<br/>/tmp に保存
    end

    rect rgb(240, 255, 240)
        Note over DEVICE: フェーズ3: 検証
        Note over DEVICE: ⚠️ サービスは稼働継続
        loop 各ファイル
            DEVICE->>DEVICE: チェックサム検証
        end
        Note over DEVICE: すべて検証OK
    end

    rect rgb(255, 255, 220)
        Note over DEVICE: フェーズ4: バックアップ
        Note over DEVICE: ⚠️ サービスは稼働継続
        DEVICE->>DEVICE: docker-compose.yml<br/>バックアップ
        DEVICE->>DEVICE: スクリプトバックアップ
        DEVICE->>DEVICE: 設定ファイルバックアップ
    end

    rect rgb(240, 220, 255)
        Note over DEVICE: フェーズ5: 適用準備
        Note over DEVICE: ⚠️ サービスは稼働継続
        DEVICE->>DEVICE: docker-compose.yml更新
        DEVICE->>DOCKER: ローカルタグ付け
        Note over DEVICE: 設定ファイル準備<br/>（まだ配置しない）
    end

    Note over DEVICE,SERVICE: ===== サービスは稼働継続（ここまで） =====
    Note over DEVICE,SERVICE: ===== ダウンタイム開始 =====

    rect rgb(255, 200, 200)
        Note over DEVICE: フェーズ6: サービス停止
        DEVICE->>SERVICE: docker-compose down
        SERVICE-->>DEVICE: 停止完了（30秒以内）
        Note over SERVICE: ❌ サービス停止
    end

    rect rgb(255, 220, 220)
        Note over DEVICE: フェーズ7: サービス起動
        DEVICE->>DOCKER: docker-compose up -d<br/>（新バージョン）
        DOCKER->>SERVICE: コンテナ起動
        Note over SERVICE: 🔄 起動中...
        SERVICE-->>DEVICE: 起動完了（120秒以内）
    end

    rect rgb(200, 255, 200)
        Note over DEVICE: フェーズ8: ヘルスチェック
        DEVICE->>DEVICE: 30秒待機
        loop 最大3回
            DEVICE->>SERVICE: GET /health
            SERVICE-->>DEVICE: 200 OK
        end
        Note over SERVICE: ✅ サービス正常
    end

    Note over DEVICE,SERVICE: ===== ダウンタイム終了 =====
    Note over DEVICE,SERVICE: ===== サービス正常稼働 =====

    rect rgb(220, 255, 220)
        Note over DEVICE: フェーズ9: 完了通知
        DEVICE->>DEVICE: バージョン情報更新
        DEVICE->>SYNC: POST /artifact-management/complete
        SYNC-->>DEVICE: 完了確認
        DEVICE->>DEVICE: 一時ファイル削除
    end

    Note over DEVICE: 更新完了
```

#### 重要なタイミング

| フェーズ | サービス状態 | 所要時間 | 失敗時の動作 |
|---------|------------|----------|------------|
| **1. バージョンチェック** | 稼働中 | 1-5秒 | 現在のバージョンで起動継続 |
| **2. ダウンロード** | **稼働中** | 1-10分 | 一時ファイル削除、起動継続 |
| **3. 検証** | **稼働中** | 5-30秒 | 一時ファイル削除、起動継続 |
| **4. バックアップ** | **稼働中** | 1-5秒 | 更新中止、起動継続 |
| **5. 適用準備** | **稼働中** | 5-15秒 | バックアップ復元、起動継続 |
| **6. サービス停止** | **停止開始** → 停止 | 10-30秒 | （強制停止して次へ） |
| **7. サービス起動** | **起動中** | 30-120秒 | **ロールバック実行** |
| **8. ヘルスチェック** | **稼働中（検証中）** | 30-60秒 | **ロールバック実行** |
| **9. 完了通知** | 稼働中 | 1-5秒 | ログ記録のみ（更新は成功） |

#### ダウンロードと適用の時間差

**重要なポイント:**
- **フェーズ2（ダウンロード）**: イメージやファイルをダウンロードするが、**まだ適用しない**
- **フェーズ3-5**: ダウンロードしたファイルを検証・準備するが、**まだサービスは稼働中**
- **フェーズ6**: **ここで初めてサービスを停止** → ダウンタイム開始
- **フェーズ7-8**: 新バージョンで起動・検証 → ダウンタイム中
- **フェーズ8成功**: ダウンタイム終了

**通常のタイムライン:**
```
00:00 - フェーズ1開始（バージョンチェック）
00:02 - フェーズ2開始（ダウンロード）← サービス稼働中
00:05 - フェーズ3開始（検証）← サービス稼働中
00:06 - フェーズ4開始（バックアップ）← サービス稼働中
00:07 - フェーズ5開始（適用準備）← サービス稼働中
00:08 - フェーズ6開始（サービス停止）← ★ダウンタイム開始★
00:08 - フェーズ7開始（サービス起動）← ダウンタイム中
00:09 - フェーズ8開始（ヘルスチェック）← ダウンタイム中
00:10 - フェーズ8成功 ← ★ダウンタイム終了★ (約2分)
00:10 - フェーズ9開始（完了通知）← サービス稼働中
00:11 - 更新完了
```

**ダウンタイム: 約1-3分**（フェーズ6開始 → フェーズ8成功）

| 障害箇所 | 影響 | 対応 | RTO/RPO |
|---------|-----|-----|---------|

| Sync API障害 | バージョン確認不可 | ローカルキャッシュで起動継続 | RTO: 0分 |
| エッジレジストリ障害 | POS高速配布不可 | ACR直接アクセスにフォールバック | RTO: 1分 |
| ネットワーク断 | 更新不可 | オフラインモードで運用継続 | RTO: 0分 |
| MongoDB障害 | 設定変更不可 | セカンダリへ自動フェイルオーバー | RTO: 30秒 |

### 10.3 バックアップ戦略

| 対象 | 頻度 | 保持期間 | 保存先 | 復旧方法 |
|-----|-----|---------|-------|---------|
| バージョン設定 | 変更時即座 | 1年 | Azure Blob Storage | API経由で復元 |
| docker-compose.yml | 更新前 | 1世代 | ローカル | ファイルコピー |
| startup.sh | 更新前 | 1世代 | ローカル | ファイルコピー |
| MongoDB | 日次 | 30日 | Azure Backup | Point-in-time復元 |
| ログファイル | 日次 | 90日 | Azure Log Analytics | クエリで参照 |
