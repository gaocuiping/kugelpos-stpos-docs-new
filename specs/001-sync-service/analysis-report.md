## 憲章準拠チェック

### プロジェクト憲章の8原則

すべての原則に完全準拠していることを確認しました：

#### ✅ I. マイクロサービス独立性

- 独立データベース: `sync_{tenant_id}` 形式
- サービス間通信: Dapr pub/sub、Service Invocation
- 共通ライブラリ: commons（ユーティリティのみ）
- 独立テスト: `services/sync/tests/`

#### ✅ II. 非同期優先 (Async-First)

- データベース操作: Motor (async MongoDB driver)
- HTTP クライアント: httpx（非同期）
- FastAPI エンドポイント: すべて `async def`
- ブロッキング操作: aiofiles、ThreadPoolExecutor で非同期化

#### ✅ III. テスト駆動開発 (TDD) ⭐ **提案1で強化**

- Red-Green-Refactor サイクル: tasks.md に明記
- テスト専用タスク: 17タスク (115タスク中 約15%)
- テストカバレッジ目標: 80%以上（重要ロジックは90%以上）
- テスト実行順序: test_clean_data.py → test_setup_data.py → 機能テスト

#### ✅ IV. イベント駆動アーキテクチャ

- Dapr Pub/Sub: `tranlog_report`, `cashlog_report`, `opencloselog_report`
- イベントスキーマ: Pydantic モデルでバージョニング
- 冪等性: `sync_status` フィールドで重複配信対応

#### ✅ V. 回復力とエラーハンドリング (Resilience)

- Circuit Breaker: 3回連続失敗でオープン、60秒後に半開状態
- Retry: 指数バックオフ、最大5回リトライ
- Timeout: HTTP 30秒、データベース 10秒
- エラーコード体系: 80YYZZ形式

#### ✅ VI. マルチテナンシー（厳格な分離）

- データベースレベル分離: `sync_{tenant_id}`
- API 検証: すべてのAPIで tenant_code 検証
- クロステナントアクセス禁止: JWT トークンでアクセス制御

#### ✅ VII. 可観測性 (Observability)

- 構造化ロギング: JSON形式、必須フィールド（timestamp, service_name, tenant_code, correlation_id）
- メトリクス: リクエスト数、レスポンスタイム、同期成功/失敗率
- 分散トレーシング: correlation_id を Dapr と連携

#### ✅ VIII. セキュリティファースト

- JWT トークンベース認証: edge_id + secret（24時間有効）
- シークレットのハッシュ化: bcrypt でハッシュ化
- 入力検証: すべてのAPI入力をPydantic スキーマで検証
- 環境変数管理: 機密データは `.env` ファイル（コミット禁止）
