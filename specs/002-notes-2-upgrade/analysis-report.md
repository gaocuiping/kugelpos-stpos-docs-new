## 補足: 主要な設計決定事項

### データモデル

- **5つの主要エンティティ**: DeviceVersion, UpdateHistory, EdgeTerminal, Manifest, PendingUpdate
- **インデックス最適化**: edge_id, tenant_id, store_code, device_typeに複合インデックス
- **状態遷移管理**: update_status, download_status, apply_statusの3段階管理

### API設計

- **3つのAPI契約**: Version Management API（5エンドポイント）、Artifact Management API（3エンドポイント）、Edge Sync Service API（3エンドポイント）
- **認証方式**: JWT（有効期限1時間）
- **エラーコード**: 80YYZZ形式（80=Syncサービス）

### アーキテクチャ決定

- **2段階更新**: ダウンロードフェーズ（Phase 1-3）と適用フェーズ（Phase 4-9）の分離
- **P2P優先度制御**: priority値（0-99）による優先順位制御、全シード失敗時のクラウドフォールバック
- **Container Registry**: Azure Container Registry Premium + Harbor for Edge
- **Blob Storage**: バージョンフォルダ構造（最低3世代保持）

### セキュリティ設計

- **認証**: edge_id + secret によるJWT発行
- **改ざん検証**: SHA256チェックサム（すべてのファイルとイメージ）
- **通信暗号化**: TLS 1.2以上必須
- **テナント分離**: データベースレベル分離（sync_{tenant_id}）
