# テストカバレッジ 85% 要件 (Test Coverage Guidelines)

Kugelpos POS バックエンドの信頼性と安定性を確保するため、すべてのマイクロサービスにおいて**最低 85% のテストカバレッジ**を厳格に要求します。

## 1. カバレッジ目標: 最低 85%

**いかなるサービスのコードベースにおいても、許容されるテストカバレッジの下限は 85% です。**

この指標は、コアとなるビジネスロジック（Service層、Usecase層、Model層、Utility関数）内の**Line Coverage（行網羅率）**および**Branch Coverage（分岐網羅率）**の両方に適用されます。

### 1.1 なぜ 85% なのか？
85% のカバレッジ率は、100% カバレッジという「収穫逓減の法則」（低価値な定型コードのテストに時間を奪われる状態）に陥ることなく、デグレ（リグレッション）に対する強力なセーフティーネットを提供します。

- **< 70%**: 特にエッジケースにおいて、未知のバグが潜んでいるリスクが非常に高い状態です。本番環境（プロダクション）へのデプロイは**不可**です。
- **70% - 84%**: 新規の非クリティカルな機能の初期開発においては許容されますが、本番リリースへのブロッカー（障害）となります。
- **85%+**: **必須基準**です。正常系（Happy Path）と、特定された異常系・エッジケースの両方が堅牢にテストされていることを示します。

## 2. CI/CD におけるカバレッジの強制

85% の要件は、GitHub Actions パイプライン (`pytest-cov`) を通じて自動的に強制（Enforce）されます。

1. **Pull Request Validation**: 全体のカバレッジを 85% 未満に低下させる、あるいは 85% 未満のカバレッジしか持たない新規ファイルを追加する PR（プルリクエスト）は、CI チェックで自動的に **FAIL (失敗)** となります。
2. **Coverage Reports**: ビルドごとに詳細な `coverage.xml` レポートが生成されます。これは PR のコメントや SonarQube 等で視覚化して確認できます。

## 3. 85% を達成・維持するための戦略

もし現在担当しているサービスが 85% の閾値を下回っている場合は、以下の戦略を活用してください。

1. **行だけでなく「分岐(Branch)」をテストする**: 特にエラーハンドリング（`try/except` ブロック）において、`if` と `else` の両方の条件がテストされていることを確認してください。
2. **テストのパラメータ化**: `pytest.mark.parametrize` は、最小限のコードで様々な入力（境界値、Null チェック等）に対して高いカバレッジを達成するための最も効率的な方法です。
3. **I/O アクセスの積極的なモック化**: カバレッジが伸び悩む原因の多くは、「到達困難なデータベースエラー」や「外部 API の障害状態」にあります。`unittest.mock` を用いて、これらの障害状態を意図的にシミュレートしてください。
4. **定型コードの除外 (実用的に)**: カスタムロジックを持たない純粋な宣言型コード（Pydantic モデルの定義や設定定数など）が、リスクを伴わずにカバレッジ率を不当に下げている場合は、`.coveragerc` を使用して除外してください。

### 3.1 `pytest.ini` の設定例

```ini
[pytest]
addopts = --cov=app --cov-report=term-missing --cov-fail-under=85
```
