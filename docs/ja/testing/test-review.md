## 一、自動化テストの正式構成と定義

```
自動化テスト（Automated Testing）
├── 単体テスト（Unit Test）
│     個々の関数・クラスを外部依存から隔離して検証
│     → unittest.mock / AsyncMock を使用
│
├── 統合テスト（Integration Test）
│     サービス間・レイヤー間の連携を実際の依存関係で検証
│     → 本プロジェクトでは明示的には未実装
│
└── シナリオテスト（Scenario-Based Test / E2E）
      業務フロー全体を HTTP クライアントで検証
      → http_client.get/post/put を使用
```

| No. | テスト種別 | 目的 | 実行速度 | バグ特定精度 | 本プロジェクトの状態 |
|-----|-----------|------|---------|------------|-----------------|
| 1 | **単体テスト** | ロジック単体の正確性保証 | 速い（ms） | 高い（関数単位） | ⚠️ 極少 |
| 2 | **統合テスト** | コンポーネント間連携の保証 | 中程度 | 中程度 | ❌ 未実装 |
| 3 | **シナリオテスト** | 業務フローの正確性保証 | 遅い（分） | 低い（どこが原因か不明） | ✅ 充実 |

---

> **【補足】本プロジェクトの3層アーキテクチャ**
>
> ```
> HTTP Request → [API層] → [Service層] → [Repository層] → MongoDB / gRPC
> ```
>
> | レイヤー | ディレクトリ | 役割 |
> |---------|------------|------|
> | **API層** | `app/api/` | HTTPリクエスト受信・パラメータ解析・Service呼出・レスポンス返却。ビジネスロジックを持たない薄い封装層 |
> | **Service層** | `app/services/` | ビジネスロジックの中核。状態検査・金額計算・バリデーション・例外処理が集中 |
> | **Repository層** | `app/models/repositories/` | MongoDB・gRPC・外部Web APIへのデータアクセスを担当 |
>
> そのため、**単体テストはロジックが集中するService層**を対象とし、**シナリオテストはHTTP経由でAPI層**の業務フローを検証する構成となっている。

## 二、テストピラミッドの理想と現状

### 理想のテストピラミッド

```
        /\
       /  \  シナリオテスト（E2E）：20%
      /    \  業務フローを保証
     /------\
    /        \  統合テスト：30%
   /          \  コンポーネント間連携を保証
  /------------\
 /              \  単体テスト（Unit Test）：50%
/________________\  ロジックの正確性を保証（最多）
```

### 本プロジェクトの現状（全体傾向は逆ピラミッド、ただしsync等は除く）

```
        /\
       /  \  シナリオテスト：34%（全体54ファイル）
      /    \
     /------\
    /        \  統合テスト：27%（全体43ファイル、syncのみ）
   /          \
  /------------\
 /              \  単体テスト：38%（全体60ファイル、うちsyncが47）
/________________\
```

**→ syncサービス単体では理想的なピラミッドだが、他サービスは深刻な逆ピラミッド構造の問題あり**

## 六、主要問題

### 問題1：テストピラミッドの偏り
全157ファイルのうち、syncサービスのテスト（90ファイル）は理想的だが、他サービスでは極端な逆ピラミッド構造（シナリオテスト依存）となっている。

### 問題2：sync以外の統合テストの完全欠如
syncサービスはAPI/DB連携テストを適切に備えるが、他サービス間（cart → report → journal → stock という Dapr pubsub チェーン等）の連携テストが存在しない。

### 問題3：単体テストの不均一な分布
sync・cart・journal 以外は単体テストがほぼゼロ。特に terminal の状態遷移バリデーションロジックなど重要箇所が未テスト。

### 問題4：異常系シナリオの欠如
全サービスにわたり、エラー系・境界値・不正入力のシナリオテストがほぼ存在しない。

### 問題5：テスト実行の順序依存
`run_all_tests.sh` で実行順が固定されており、テストが独立して実行できない。CI/CD での並列化が不可能。

## 七、改善提案

本セクションでは、プロジェクトの課題を解決し、理想的なテスト構造へ移行するための具体的なアクションプランの詳細方針を提示します。

### 方針1：テストピラミッドのバランス回復

現在過剰になっているシナリオテストへの依存を減らし、単体テストと統合テストの比率を高めることで、実行速度・保守性・信頼性のバランスが取れたテスト基盤を構築します。

| No. | 目標 | 現状 | 目標値 |
|-----|------|------|--------|
| 1 | 単体テスト比率 | **38%** | 50% 以上 |
| 2 | 統合テスト比率 | **27%** | 30% 以上 |
| 3 | シナリオテスト比率 | **34%** | 20% 程度 |

---

### 改善アクションプラン詳細

以下に、優先度順（P0～P3）に整理した具体的な改善シナリオとタスクの詳細を説明します。表形式での羅列から、実際の開発者がどう取り組むべきかの詳細な見解を記載します。

#### 【最優先: P0 緊急】致命的リスクの排除と基盤の安定化

**1. 単体テストの追加構築（対象：terminal, master-data）**
*   **現状の課題**: `terminal` や `master-data` サービスでは単体テストが完全に欠落しており、ビジネスロジックの変更が他の機能に予期せぬ不具合（デグレード）を及ぼすリスクが極めて高い状態です。
*   **具体的手順**: 
    1.  **依存関係の排除**: ビジネスロジックが集中する `app.services` 配下のクラスをテストする際、実際のデータベース通信（Repository層）や外部通信（gRPC等）を避けるため、`pytest-mock` を用いてこれらのクラスやメソッドを `AsyncMock` でスタブ化（擬似化）します。
    2.  **状態遷移バリデーションの検証**: `terminal` などの重要な状態を管理するサービスにおける「開局・閉局ステータスチェック処理・遷移ロジック」を重点的に検証します。異常なステータス変更操作の際に、適切なカスタム例外（例: `HTTPException(400)`）が送出されるかどうかをテストメソッドで明示的に定義します。
    3.  **データ駆動（パラメータ化）テストの活用**: 金額計算（内税・外税の扱いの違い等）などの分岐条件が多い箇所は `@pytest.mark.parametrize` アノテーションを使用し、すべての境界値パターンに応じた網羅的なアサーションを実施します。
*   **想定工数 / 期待効果**: 5営業日 / ロジックの品質保障を強固にし、本番環境へのビジネスロジックにおける致命的バグの流出を未然に防止します。

**2. Dapr 受信シナリオテスト追加（対象：journal の tranlog受信API）**
*   **現状の課題**: 非同期メッセージング（Dapr PubSub）を利用したジャーナルデータの登録処理に対して、十分なシナリオテストが無く、パースエラーによるログ喪失の運用リスクが存在します。
*   **具体的手順**:
    1.  **PubSubモック環境の構築**: Daprの `Testcontainers` (またはインメモリモック機能) を利用して、テストランナー側から直接 `translog_topic` トピックに対してダミーイベントを送信（パブリッシュ）できる結合用のテスト環境を整備します。
    2.  **非同期処理フローの検証**: 送信したJSON形式のペイロードが正しく解析・変換され、バックエンドの MongoDB コレクション（または同等のモックDB環境）に適切なスキーマで Insert/Update されるかを確認します。非同期通信による処理遅延を考慮するため、`await asyncio.sleep(0.1)` などの十分な待機時間を設けたのちに `find_one` で最終的なデータ状態をダイレクトにアサートします。
*   **想定工数 / 期待効果**: 3営業日 / 例外ハンドリングの漏れを塞ぎ、データ喪失・トランザクション不整合といった致命的な障害を未然に防ぎます。

#### 【高優先: P1 構造/網羅】システム全体の整合性保証とカバレッジ回復

**3. Dapr PubSub 統合テスト実装（対象：cart → report → journal → stock の非同期連携）**
*   **現状の課題**: 各サービスの単独動作のみで、実際のシステム動作の中心である非同期処理チェーン全体の順序性保証（結合レベル）ができていません。
*   **具体的手順**: 
    1.  **インフラのコンテナ化**: `docker-compose.test.yml` を新規に作成し、本番環境と同一バージョンの MongoDB, Redis, Dapr ネットワーク構成を立ち上げ、統合テストの実行ベースとなるクリーンな環境を作ります。
    2.  **エンドツーエンド (E2E) フローの自動化**: `cart` の決済処理完了APIを呼び出します。これにより起爆されたイベントが、`report` (売上集計)、`journal` (ジャーナル履歴の保存)、`stock` (在庫引当) までの連鎖を正常に経由しているかを検証します。これについてはそれぞれのサービスのデータベースを検証しアサートします。
*   **想定工数 / 期待効果**: 8営業日 / 単体機能としてではなく、非同期的な結果整合性をシステム全体で網羅・保証できるようになります。

**4. `item-book`シナリオ網羅（対象：master-data の15パス・12API）**
*   **現状の課題**: `master-data` 内で中核をなす `item-book` 関連エンドポイントのAPIシナリオテストが不足しています。
*   **具体的手順**: 
    1.  **エンドポイント呼び出しの網羅**: 全12のAPIエンドポイントに対し、HTTPクライアントを使った作成→取得→更新→削除（CRUD）のシナリオを一通り組み込みます。
    2.  **組合せ(コンビネーション)テストによる精度向上**: 商品のバリエーション（税区分A/B、各種設定フラグ等）を全網羅したデータ配列をパラメータとし、各々の `GET /items` レスポンス内容の正当性について、JSONスキーマバリデータ（`jsonschema` ライブラリ等）を利用してフィールドレベル（名称、金額、SKU）の完全一致を検査します。
*   **想定工数 / 期待効果**: 2営業日 / `master-data` におけるクリティカルパスのテストカバレッジが急速に回復します。

**5. 異常系シナリオテスト追加（対象：全サービス横断）**
*   **現状の課題**: 既存テストが「正常系（Happy Path）」を想定してのみ作られているため、バリアントデータやインフラ障害に対する耐性が評価されていません。
*   **具体的手順**: 
    1.  **4xx系（クライアントエラー）ハンドリングの網羅**: 必須項目が欠落したリクエスト、型違反エラー (Pydanticバリデーションエラー)、無効なIDに対する操作やアクセス（HTTP 404）、業務の制約に反した操作ルール違反（例：既に処理済の決済IDに対する二重リクエスト - 400や409の再現）、アクセス権が無い領域へのテスト（HTTP 403）などの攻撃を模したスキャン実行を追加します。
    2.  **5xx系（サーバーエラー）ハンドリングの網羅**: データベースサーバーとの切断や、連携先システムの通信タイムアウト発生時に、アプリケーション自体が強制終了せず、フロントに対し例外情報を適切に包んだ `500 Internal Server Error` がきちんと返却されるかを検証します。
*   **想定工数 / 期待効果**: 5営業日 / 予期せぬ入力や境界条件への耐障害性（レジリエンス）、外部アタックへのセキュリティ安全性が向上します。

#### 【中優先: P2 基盤】テスト基盤の効率化と品質管理の自動化

**6. 実行依存の解消とデータファクトリ導入（対象：全シナリオテスト）**
*   **現状の課題**: 現行のテストランナー（`run_all_tests.sh` 等における順次実行）が既存データの状態に強く依存しており、CIでの並行処理の阻害やFlaky（結果が不安定となる）なテストになっています。
*   **具体的手順**: 
    1.  **データジェネレータツールの導入**: `factory_boy` 又は `faker` を用いて、個別のテストに必要な「ユニークな文字列を持つテナントID」「初期残高等を持つ店舗マスタ」のデータを、ハードコーディングせず動的に都度生成する仕組みを作ります。
    2.  **テスト状態分離の基盤構築（クリーンアップ）**: テスト初期化スクリプトである `conftest.py` に `yield` を用いて、各テストの `setup` (データ準備) と `teardown` (データクリーンアップ) の実行を強制します。これによりDBの初期状態が担保でき、最終的に `pytest-xdist` を利用した並行起動 (`pytest -n auto`) の前提を整えます。
*   **想定工数 / 期待効果**: 5営業日 / 安定するテストの実現と、並行起動によりCI/CDのフィードバックループ（実行時間）を劇的に短縮させます。

**7. カバレッジ門限（Quality Gate）の設定（対象：全プロジェクト）**
*   **現状の課題**: プロジェクトに「どの程度テストを書くべきか」という合意形成ルールがCI化されていません。
*   **具体的手順**: 
    1.  **自動化実行時のパイプライン連携**: GitHub Actions 等のCI環境で実行しているテストコマンドラインへ `--cov=app --cov-report=xml --cov-fail-under=70` 等を追記し、コードカバー率が指定値（例:70%）を下回る場合はそのビルドを失敗させる運用を実施します。
    2.  **レポートの視認性向上とブロック条件**: Codecov や SonarQube との連携を確立し、プルリクエストに対する自動レビューコメントを付与します。カバレッジ劣化が見られる場合など、品質チェック基準を満たしていない時はメインブランチ側へのマージをブロックします。
*   **想定工数 / 期待効果**: 1営業日 / CIによる自動監視を取り入れ、長期的かつ継続的なコード品質劣化ルールを防御します。

#### 【将来課題: P3 基盤】高可用性のプロアクティブな担保

**8. 負荷テスト体制の導入（対象：主要ビジネスフローに対する非機能要件）**
*   **現状の課題**: コア業務処理が集中する時間帯での最大性能や限界到達リソース値の検証が抜け落ちています。
*   **具体的手順**: 
    1.  **シナリオスクリプトの記述（TaskSet生成）**: Python言語で構築可能な負荷テストツール `Locust` などをプラットフォームとし、代表的な業務フローである「ログイン → カート生成 → 商品バーコードスキャン → 支払い → 決済」という挙動を設定したユーザーシナリオスクリプトを記述します。
    2.  **性能基準（ベースライン）の策定**: まず本番環境と等価なステージング環境に対し、たとえば 500 RPS (Requests Per Second) という一定の負荷を10分間継続的に与え続けます。この条件のもとで、95パーセンタイルのレスポンス時間が実務上支障のない「2秒未満」にとどまっているか、またサーバー群のCPU・メモリ利用率からコネクションプールの状況といったシステムの体力が正常維持できているか計測、レポート化します。
*   **想定工数 / 期待効果**: 5営業日 / リリース前における高負荷時の性能限界値の可視化とボトルネック箇所特定の体制を構築し、可用性を堅牢に維持します。

---

## 八、総合結論

本プロジェクトの自動化テスト体系は、**syncサービスにおいて理想的なテスト構造（単体テスト・統合テストの充実）を実現している**ものの、**他の主要サービス（cart, terminal, master-data等）はシナリオテストに過度に依存**しており、不均一なテスト戦略となっている。これは以下のリスクをはらんでいる：

1. **デグレード検知の遅延**：sync以外のサービスは単体テスト・統合テストが少ないため、ロジック変更のバグが発見しづらい。
2. **サービス間連携の盲点**：各サービスを跨ぐ pubsub メッセージ処理チェーンの実装において結合レベルの保証がない。

**最優先で取り組むべき課題**は、syncサービスの優秀なテスト自動化プラクティス（統合テストと単体テストの分離構成）を、他サービス（特に terminal や master-data）への横展開し、プロジェクト全体のテストピラミッドを底上げすることである。
